## Content writing style (applies to all playbook prose)

All prose in this playbook must follow the universal **Writing Style** rules in `~/.claude/CLAUDE.md`.
That file is the source of truth. Do not duplicate it here.
The rules that bite most often when editing pages:

- Write in a natural, experienced-engineer voice. Not marketing copy.
- Never use the em dash "-". Use a plain dash "-" instead. This includes decorative labels like page numbers.
- Do not use semicolons to join two ideas. Use separate sentences.
- Do not compress several ideas into one sentence, and avoid compressed parenthetical lists such as "(batch, sample, redact, enrich)". Break them into short sentences or bullets.
- Keep sentences short and direct.
- Keep commands and config in code blocks, in monospace, never in prose or handwriting.

This is part of the process for every content change, not just the notebook pilot.

## Site structure & sharp edges

This is a **multi-page static site** (no build step): `index.html` is ONLY the bookshelf landing, and each topic is its own file (`workstation-setup.html`, `ai.html`, `kubernetes.html`, `helm.html`, `argocd.html`, `mcp.html`, `opentelemetry.html`, `aws-adot.html`). `styles.css` and `script.js` are SHARED - every page links the same two files; the engine/styles are never duplicated per page. The topic pages were generated by splitting the old monolith; if you need to regenerate them the split script lived in the session scratchpad (`split.py`, reads a `monolith.html` backup). To EDIT one topic now, just edit its file - they are independent.

- **Assets are cache-busted with a `?v=N` query** on EVERY page's `styles.css`/`script.js` links. When you change `styles.css` or `script.js`, bump the number on **all** HTML files together (they must stay in lockstep) or browsers serve stale cached assets. A quick `perl -i -pe 's/\?v=OLD/?v=NEW/g' *.html` keeps them consistent.
- **Theme + plain-text persist across page navigation.** A tiny inline `<script>` in each page's `<head>` reads `localStorage` (`theme`, `nbPlain`) and applies `data-theme` + `html.nb-plain` **before paint** (no flash). `setTheme()`/`toggleNbPlain()` write those keys. Never rely on in-memory theme state surviving a navigation - it does not.
- **Shared `script.js` runs on every page**, so all top-level DOM access is guarded (missing `#sidebar`/`#hamburger`/`#progFill`/`#particleCanvas`/canvases must not throw). `initNbBooks()` returns early when there is no `.nb-book` (the shelf), `initShelf()` returns early when there is no `#shelf` (topic pages). If you add top-level code that touches an element, guard it or you break every page that lacks it.
- **Legacy note (pre-split):** the old single-page layout used `<section class="section">` + a sidebar `.nav-link[data-section]` list with two IntersectionObservers for scroll-spy. That sidebar is gone. Breadcrumb is now driven directly by the book controller from `data-topic` (on the book section) + the current leaf's `data-label`.
- **Sections** are `<section id="..." class="section">` (older ones use `content-section`) appended inside `<main>`.
  Two IntersectionObservers in `script.js` auto-wire this for every `.section`/`.content-section` - no registration needed. Just add the section and a matching `.nav-link[data-section=...]`.
  - **Scroll-spy is height-independent by design - keep it that way.** Active-section detection uses a **trigger-band** observer (`threshold: 0`, `rootMargin: "-45% 0px -45% 0px"`), so whichever section crosses the thin band near the viewport middle is the active one regardless of section height. Do NOT revert to a visibility-ratio threshold (e.g. `threshold: 0.2`): OTel sections like Core Concepts / Traces, Metrics & Logs are 6-7x viewport height and can never reach a positive ratio, so a ratio threshold silently never marks them active (nav/breadcrumb/progress stall). `pickActiveSection()` picks exactly one (topmost if two straddle the band; scroll-position fallback at page top/bottom) via the shared `setActiveSection(id)` helper. Reveal + progress use a separate `threshold: 0` observer so tall sections still count toward progress and get `.visible`.
- **Nav groups** are collapsible `<li class="nav-group" id="...">` toggled by `toggleNavGroup(id)`.
- **Sticky top bar / breadcrumb** (`.topbar`, first child of `<main>`) shows `group / section` and hosts the theme toggle. The breadcrumb (`#tbGroup`/`#tbSection`) is updated by the shared `setActiveSection()` that drives scroll-spy - it mirrors the active nav-link's label and its `.nav-group` label. Sections carry `scroll-margin-top` so anchor jumps clear the bar; keep that if you touch section padding. The topbar is the one intentional use of `backdrop-filter` (frosted sticky header) - not glassmorphism-by-default.
- **CSS design tokens** live in `:root` (dark) and `[data-theme="light"]` at the top of `styles.css`. Both themes are first-class and fully defined; drive everything off `var(--*)` so a theme is a pure token swap. Never hardcode a text/background color that must survive a theme flip - use the ink ramp (`--txt`/`--txt2`/`--txt3`), surfaces (`--bg`/`--bg2`/`--bg3`/`--surface-1..3`), borders (`--border`/`--border-h`), brand (`--brand`, `--accent`, `--accent-ink`, `--accent2`), or semantic (`--blue`/`--green`/`--orange`/`--red`/`--cyan`). The semantic + section-brand tokens (`--helm`/`--argocd`/`--mcp`/`--adot`) are re-declared under `[data-theme="light"]` to darker shades so they pass contrast as text on light; keep that pattern for any new brand color.
  - Legacy `--glass*` names are now **solid surface aliases** (not translucent glass) - the design is restrained, not glassmorphic. No `backdrop-filter`, no `background-clip:text` gradient text, no `>1px` colored side-stripe borders (use full 1px borders + tint + a leading icon for callouts; neutral tree-guide left-borders are fine). Card radius stays 12-16px.
  - **`--code-well`** is the token for terminal/sim/code panels: a dark inset in dark mode, a light inset in light mode. Text inside inherits theme ink, so it stays legible in both. The macOS-style `.term-*` terminal is the exception - it stays dark in both themes and uses fixed (non-token) colors on purpose.
  - Motion uses `--ease` (ease-out) and there is a global `prefers-reduced-motion` block; section reveals are `animation ... both` so content is never gated on a JS class (never ships blank on hidden tabs/headless).
  - Responsive safety net near the code-block rules: code surfaces get `overflow-x:auto`/`min-width:0`, and grid/flex children get `min-width:0` so wide code in sim cards scrolls internally instead of forcing horizontal page overflow on mobile.
- Reuse the existing component kit (`.sec-header`, `.card`, `.eli5-card`, `.ctag`, `.two-col`, `.card-grid-4`, `.timeline`, `.devops-box`, `.takeaways`, `.callout` with `.callout-icon`, `.code-block`/`.cb-code` with `.kw`/`.str`/`.cmt`/`.num` highlight spans) rather than inventing styles. Namespace any new JS/CSS per-feature (e.g. `otel*`, `adot*`).
- **Native theme-aware diagrams** (no raster screenshots): build them with inline SVG or CSS grid and drive every color off tokens so they flip with the theme. Reusable patterns already exist in the OTel deep-dive sections: `.otel-seq*` (SVG sequence diagram - lifelines, activation bars, request/response arrows via `<marker>`, colors set inline as `fill:var(--blue)` etc.), `.otel-wf*` (Jaeger-style trace waterfall in CSS grid - bars positioned with inline `left`/`width` percentages, indented per depth), `.otel-cpipe`/`.ocp-*` (a horizontal box→box→box pipeline flow with a per-box `--ocp` color var and arrows that rotate to vertical on mobile - used for the Collector Receiver→Processor→Exporter and app→agent→gateway diagrams), `.otel-anat*` (a labeled field-by-field "anatomy" table), and `.otel-comp*` (mono component chips + explanation lists). Wrap any wide SVG in an `overflow-x:auto` scroller with a `min-width` so it scrolls internally on mobile instead of overflowing the page.
- **The whole site is now notebooks (`nb-*` module).** Every top-level topic is one page-turn "book": Workstation Setup, AI, Helm Charts, Argo CD, MCP, AWS ADOT, OpenTelemetry, and Kubernetes Notes. Each is ONE `<section id="nb-<topic>-cover" class="section nb-section nb-book-section nb-book--<topic>">` whose children are `.nb-leaf` panels (a cover, the topic's pages, an end/back-cover leaf) inside `.nb-book > .nb-stage`.
  - **Two content models.** Kubernetes Notes is authored in the handwriting notebook primitives (`.nb-sheet.nb-cover`/`.nb-page`, `.nb-h`/`.nb-p`/`.nb-box`/`.nb-table`/`.nb-code`/`.nb-flow`). The other seven topics were PORTED AS-IS: their original sections are wrapped in `.nb-sheet.nb-page.nb-ported` (paper + spiral binding + masthead + page number) but the content keeps its own site styling, brand colours, simulations, and Copy buttons untouched. Do not rewrite ported content into handwriting; keep it as-is.
  - Every leaf still has `.nb-rings` capsules over a gutter `::after`, a red margin line (`::before`), and ruled paper (a `repeating-linear-gradient`). Per-book brand tint: `.nb-book--<topic>` recolours only the notebook chrome (`--nb-marker` drives cover title, masthead, controls) to the topic's brand token, so ported content and chrome both feel like that topic.
  - **Page-turn engine is one shared controller, `initNbBooks()` in `script.js`.** It scans every `.nb-book`, adds `.nb-ready`, and drives each independently: one leaf at a time in a fixed reading frame with 3D `rotateY` turns (book-open swing on the cover, book-close swing onto the end leaf). Without JS the leaves fall back to a normal vertical stack that scrolls, so **content is never gated on the animation**. Navigation: Prev/Next buttons + dog-ear corners + keyboard Left/Right (only while that book fills the viewport) + touch swipe. Each leaf scrolls INTERNALLY (`overflow-y:auto`); only movement BETWEEN leaves is a turn. Controls are found by class within each book (`.nb-prev`/`.nb-next`/`.nb-page-indicator`/`.nb-corner-*`), never by id.
  - **Scroll-spy/breadcrumb/deep-links:** leaf ids equal the sidebar `data-section` ids (the ported topics kept their original bare ids like `otel-intro`; covers and the Kubernetes pages use the `nb-<topic>-...` form). `indexOfId()` normalises by dropping a leading `nb-`, so a deep link resolves in EITHER convention - both `#nb-otel-intro` and `#otel-intro` open the otel intro leaf. Any `#<id>` deep-link opens the owning book directly on that leaf. The controller keeps `window.nbCurrentPage` (sectionId -> current leafId) in sync and `setActiveSection()` redirects a book section id to its current leaf so the breadcrumb/nav track the page on screen. `render()` only writes the URL hash on user-driven turns (init passes `render(false)` so books do not clobber a deep link). Nav-link clicks and every `hashchange` (address bar, back/forward, cross-book links, `#shelf`) route through `window.nbBookGoto()`, which finds the OWNING book, scrolls its section into view, and opens it at the right page (open-swing for a cover). This is what makes crossing from one book (or the shelf) to another actually switch books - do not revert the hashchange handler to calling a single book's `go()`, which never scrolls to the target book.
  - Reduced motion: a `@media (prefers-reduced-motion: reduce)` block drops the leaf `rotateY` to `none` and the global reduced block zeroes transitions, so turns become an instant switch; the controller also sets its finalize delay to 0.
  - Both themes are token-driven via `--nb-*` declared in `:root` (dark = "night notebook", dark paper) and re-declared under `[data-theme="light"]` (bright cream paper). Never hardcode a notebook color that must survive a theme flip.
  - Fonts: Permanent Marker (cover title), Caveat (headings), Kalam (body), loaded from the shared Google Fonts link in `index.html`. `html.nb-plain` is the **accessibility plain-text toggle** - it swaps the three handwriting vars for the site sans. `toggleNbPlain()` in `script.js` persists the choice in `localStorage` under `nbPlain`.
  - **Code stays real monospace, always.** Kubernetes kubectl blocks use `.nb-code > pre`; ported topics keep their existing code blocks and `.term-copy` Copy buttons. Wide content (tables, diagrams, code) gets `overflow-x:auto` so it scrolls internally on mobile.
  - **Page width is content-driven (no dead ruled area).** The paper is a book-width page centred on the desk (leaf transparent, `--nb-desk` on the stage). `initNbBooks()` tags each content leaf `nb-w-text` (narrow, readable) or `nb-w-wide` (fills the page so two-column content sits side by side) by scanning for grids/tables/diagrams/canvas. Do not reintroduce a fixed left-half `max-width` cap on ported content.
  - **In-book nav = bookmark ribbon rail (`.nb-bookmarks`/`.nb-bm`), built by `initNbBooks()`** from the (hidden) sidebar nav labels - one tab per leaf plus a back-to-shelf link. Right-gutter rail on desktop, sticky horizontal strip under the top bar on mobile. Tabs are `role="tab"` buttons, reflect the current page (`.is-active`/`aria-selected`), and call `go(i)`. `render()` keeps them synced.
- **No global sidebar.** The left sidebar is hidden (`display:none`) and `.main` is full-width; the bookshelf is the home nav and the bookmark rails are the in-book nav. The `#navList` links stay in the DOM only so `setActiveSection()` (breadcrumb) and `nbBookGoto()` can resolve page ids and labels.
- **Bookshelf landing = `index.html`** (only the shelf, no notebooks). It is a cozy library of the 8 notebooks as books on wooden shelves (`.shelf-row` = one shelf, `.shelf-book` = one topic). **Grouping lives in the HTML** - regroup by moving a `.shelf-book` to another `.shelf-row`. Each book is a real link to its topic page (`<a href="helm.html">`), keyboard focusable (Enter native, Space added in `initShelf()`), coloured by its topic brand token via inline `--book`. No search box. Book-cover titles are kept small enough (`--font-display` 0.72rem) that the longest single words fit, so they wrap at spaces (not mid-word); "Open Telemetry" and "Workstation Setup" wrap to two lines. Wood (light) vs night-library (dark) is a pure `--shelf-*` token swap; hover lift respects `prefers-reduced-motion`.
  - **The shelf fits the viewport (ADDENDUM 8):** at `min-width:760px` and `min-height:680px` the shelf is a flex column of height `calc(100dvh - 58px)` with `overflow:hidden`; rows are `flex:1`, books are `height:100%` + `aspect-ratio:112/150` so all shelves show at once with no scroll. Below those breakpoints the block is off and the shelf scrolls normally (graceful fallback) - never clip. There is no footer on `index.html` (it would push past 100dvh).
- **Topic page = one notebook.** Each topic file is the shared shell (topbar breadcrumb + theme toggle) plus that one `<section class="nb-book-section" data-topic="...">`, and links shared `styles.css`/`script.js`. No site `<footer>` (it would push the page-turn footer off-screen). `initNbBooks()` finds the single `.nb-book`, plays the book-open swing on load (cover in view), and drives the bookmark rail + page-turn. The breadcrumb starts `📚 Shelf` (a link back to `index.html`) then the topic then the current sub-page; the rail's `← Shelf` is also a link to `index.html`. Sub-page deep-links (`opentelemetry.html#otel-collector`) resolve on cold load via the `nb-`-normalising `indexOfId()`.
  - **The open book fits the viewport (ADDENDUM 9):** at `min-width:861px` + `min-height:680px` the `.nb-book-section` is `height: calc(100dvh - 58px); overflow:hidden` (a block, so the book stays centred via `margin:0 auto` - do NOT make the section a flex container or auto-margins collapse the book to content width). The `.nb-book` is a flex column at `height:100%`; the `.nb-stage` is `flex:1` (fills the space between top bar and the Prev/Cover/Next footer) and a tall page scrolls INSIDE its leaf (`.nb-leaf overflow-y:auto`), so the footer + bookmark rail never scroll off. `.nb-book` max-width is 1040 (book-like, centred on the desk); the sheet width tiers (`nb-w-text` 680 / default 860 / `nb-w-wide` 1000, clamped to the stage) give a comfortable page with two columns where content suits it. Below those breakpoints the page scrolls normally (graceful fallback).
