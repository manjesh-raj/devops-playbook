Module 03

AI Agents — Deep Dive

From simple loops to self-correcting, planning, multi-step reasoning agents

🟢 Beginner → 🔵 Conceptual → 🟠 Practical → 🔴 Architecture
👶 ELI5

A regular LLM is a brilliant person in a sealed room — they can answer questions but can't act. An AI Agent is the same person with a phone, a laptop, and the ability to use tools. They can search the web, run bash commands, call Kubernetes APIs, read log files, and send Slack messages. They accomplish goals, not just answer questions — and they loop until the task is done.

🏗️ Four Major Agent Architectures

Each architecture is suited for different complexity levels and tasks:

ReAct

Reasoning + Acting — The Foundation

ReAct interleaves reasoning traces (thinking out loud) with tool actions. The LLM writes its thought process, picks a tool, gets the result, thinks again. This loop continues until the answer is found. It's the most common agent pattern.

💭Thought

LLM reasons about what to do next

→
⚡Action

Calls a tool with specific input

→
👀Observation

Receives tool output back

↺ repeat
📜 Real ReAct Trace (K8s Incident)
💭 Thought: I need to investigate why the api-server pod is restarting. Let me check its logs first.
⚡ Action: run_kubectl("logs api-server-9xf2k --previous -n production")
👀 Observation: "FATAL: Killed process 1 (java) total-vm:2097152kB, anon-rss:256120kB — OOMKilled"
💭 Thought: The pod was OOMKilled. Memory limit is too low. Let me check what the current limit is.
⚡ Action: run_kubectl("get pod api-server-9xf2k -o jsonpath='{.spec.containers[0].resources}'")
👀 Observation: {"limits": {"memory": "256Mi"}, "requests": {"memory": "128Mi"}}
💭 Thought: Memory limit is 256Mi but process was using 256MB+ before being killed. Need to increase limit to at least 512Mi. I'll patch the deployment.
⚡ Action: patch_deployment("api-server", {"memory_limit": "512Mi"})
👀 Observation: "deployment.apps/api-server patched successfully. Rollout in progress."
💭 Thought: Fix applied. I should verify the pod is now running and create a ticket.
⚡ Action: create_ticket("OOMKilled: api-server memory limit increased 256Mi→512Mi. Root cause: JVM heap growth.")
✅ Final Answer: Root cause identified: OOMKilled due to insufficient memory limit (256Mi). Applied fix: increased to 512Mi. Pod is now Running. Ticket PLAT-2849 created.
Use ReAct when: You have a clear goal, the task needs tool use, and the steps aren't fully predictable in advance. Limitation: Can get stuck in loops or make mistakes without self-correction.
Reflection

Generate → Critique → Revise Loop

A Reflection agent has two roles in one (or two separate LLM calls): a Generator that produces output, and a Critic that evaluates and suggests improvements. The loop runs until the output meets quality standards.

✍️Generate

Produce initial output

→
🔍Critique

Self-evaluate quality

→
✏️Revise

Improve based on critique

↺ until quality met
📜 Reflection Agent: Runbook Writer
✍️ Generate (attempt 1): "If pod crashes, restart it."
🔍 Critique: This runbook is too vague. Missing: how to diagnose the crash reason, specific kubectl commands, escalation path, monitoring steps. Grade: 3/10
✏️ Revise (attempt 2): Added kubectl commands, error type matrix, escalation steps...
🔍 Critique: Better. Still missing: rollback procedure, impact assessment, post-mortem template. Grade: 7/10
✏️ Revise (attempt 3): Full runbook with all sections...
🔍 Critique: Comprehensive. All sections present. Clear commands. Good escalation. Grade: 9/10 — acceptable.
✅ Output accepted after 3 reflection cycles. Quality score: 9/10
Use Reflection when: Output quality matters more than speed. Content generation, code review, documentation writing. DevOps use: Auto-generating runbooks, post-mortems, architecture decision records (ADRs).
Reflexion

Reflection + External Grounding + Memory

Reflexion extends Reflection by adding external evaluation (running tests, checking metrics) and episodic memory — the agent remembers what failed before and avoids repeating mistakes across multiple attempts.

🎭Actor

Generates response using memory of past failures

→
🧪Evaluator

Tests output externally (runs tests, checks accuracy)

→
💡Self-Reflect

Analyses failure, writes lesson to memory

↺ retry with memory
📜 Reflexion Agent: Terraform Fix
🎭 Actor (attempt 1): Writes Terraform to create S3 bucket with wrong region format
🧪 Evaluator: terraform validate → Error: "region" must be string like "us-east-1", not "us-east". Test FAILED.
💡 Self-Reflect: I used wrong region format. Lesson stored in memory: "AWS region must include full name e.g. us-east-1 not us-east"
🎭 Actor (attempt 2, with memory): Rewrites Terraform with correct region — also recalls another memory: "S3 bucket names must be globally unique"
🧪 Evaluator: terraform validate → ✅ Success. terraform plan → ✅ Creates 1 resource.
✅ Passed after 2 attempts. Memory updated with 2 lessons for future use.
Use Reflexion when: Tasks have external feedback (tests, compilers, APIs). Agent needs to learn across attempts. DevOps use: IaC generation with validation, script writing with test suites, config generation.
Plan-and-Execute

Think First, Then Act — More Efficient Than ReAct

Instead of deciding the next action one step at a time (like ReAct), a Plan-and-Execute agent creates a full plan upfront using a Planner LLM, then executes each step. A Replanner re-evaluates if something unexpected happens.

📋Planner

Generate full multi-step plan from goal

→
⚡Executor

Run each step of the plan

→
🔄Replanner

Adapt plan if step failed or new info found

📜 Plan-and-Execute: Cloud Cost Investigation
📋 Planner: Goal is "reduce AWS costs by 20%". My plan: Step 1: Get cost breakdown by service (Cost Explorer API) Step 2: Identify top 3 cost drivers Step 3: For each: analyse usage patterns (CloudWatch) Step 4: Generate specific optimisation recommendations Step 5: Create Terraform PR with changes
⚡ Execute Step 1: cost_explorer.get_cost(last_30_days) → EC2: $12k, RDS: $3k, S3: $900
⚡ Execute Step 2: Top 3: EC2 (72%), RDS (18%), S3 (5%)
⚡ Execute Step 3: CloudWatch CPU avg → 14 instances under 5% utilisation
🔄 Replanner: Step 3 found 14 idle instances (unexpected). Updating plan: Add step 3b — verify instances aren't needed for scheduled jobs before marking for termination.
⚡ Execute Step 3b: Check CloudWatch scheduled tasks → 2 instances needed for nightly backups, 12 can be terminated
✅ Plan complete with one replan event. Savings identified: $4,200/month. Terraform PR created.
Use Plan-and-Execute when: Tasks are long, complex, and have many steps. Better cost efficiency vs ReAct — Planner uses expensive model once, Executor uses cheaper model for each step. DevOps use: Multi-phase incident investigation, complex change management workflows.

🚨 Failure ModesWhat Nobody Shows You

When Agents Go Wrong — And How to Handle It

Real agents fail constantly. Here's every failure mode and the production fix.

Failure ModeWhat HappensProduction Fix
Infinite loop Agent keeps calling the same tool, never reaches a conclusion Set recursion_limit=25 in LangGraph — raises GraphRecursionError after N steps
Tool error kubectl returns non-zero exit, Prometheus returns 500 — agent sees raw error Wrap tools with try/except, return structured error: {"error": "pod not found", "suggestion": "check namespace"} — LLM can reason on it
Hallucination LLM invents a pod name, namespace, or metric that doesn't exist Always validate tool input against real K8s resources before executing. Add a Reflexion critique node.
Wrong tool chosen Agent calls restart_deployment when it should have called get_logs first Add Human-in-the-Loop approval gate before any destructive action. Use LangGraph interrupt().
Context window overflow Long log files + tool history fills the 128K context — API returns 400 error Summarise tool outputs before adding to state. Use trim_messages() to prune old history.
Rate limiting 100 parallel incidents hit OpenAI API limit — 429 Too Many Requests Add exponential backoff + jitter. Use semaphore to cap concurrent agent threads.

Try: kubectl get pods -n {namespace}" return result.stdout[:4000] # trim to avoid context overflow except subprocess.TimeoutExpired: return "ERROR: kubectl timed out after 10s. Cluster may be unreachable."

Tools & Tool Calling

The hands of an AI agent — how LLMs take real actions in the world

🟢 Beginner → 🔵 Conceptual → 🟠 Practical → 🔴 Architecture
👶 ELI5

A tool is just a Python function the AI is allowed to call. The AI sees the function's name, description, and what parameters it takes — but not the code. When it decides to use a tool, it says "call query_prometheus with argument 'cpu_usage > 90'". Your system runs the function and sends the result back to the AI.

🔩 Anatomy of a Tool — The Schema

This is what the LLM actually sees for each tool (JSON Schema):

PythonDefine a Tool
from langchain_core.tools import tool

@tool
def query_prometheus(query: str) -> str:
    """Query Prometheus metrics API.

    Args:
        query: A valid PromQL expression
    Returns:
        The metric result as a string
    """
    result = prometheus.query(query)
    return str(result)
JSON SchemaWhat the LLM sees
{
  "name": "query_prometheus",
  "description": "Query Prometheus metrics API.
    Args: query: A valid PromQL expression
    Returns: The metric result",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "A valid PromQL expression"
      }
    },
    "required": ["query"]
  }
}
💡 The description is critical — it's how the LLM decides when to use this tool. Write clear, specific descriptions with examples of when to use the tool.

from langchain_core.tools import tool

# 1. Define your DevOps tools
@tool
def query_prometheus(query: str) -> str:

    return prometheus_api.query(query)

@tool
def run_kubectl(command: str) -> str:
    """Execute a kubectl command. Always use --dry-run for destructive operations."""
    return subprocess.run(["kubectl"] + command.split(), capture_output=True).stdout

@tool

    """Semantic search over internal runbooks and playbooks."""
    return vector_store.similarity_search(query, k=3)

💻 What Multi-Agent Code Actually Looks Like

A supervisor routes to specialist agents — each is its own LangGraph create_react_agent

PythonSpecialist agents
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import create_react_agent

llm = ChatOpenAI(model="gpt-4o-mini")

# Each specialist has its own tools
metrics_agent = create_react_agent(
    llm, tools=[query_prometheus, get_cpu_usage]
)
logs_agent = create_react_agent(
    llm, tools=[get_pod_logs, search_log_patterns]
)
remediation_agent = create_react_agent(
    llm, tools=[restart_deployment, scale_replicas],
    # Only this agent can change things
)
PythonSupervisor routes between them
from langgraph.graph import StateGraph, END

def supervisor(state):
    # LLM decides which agent to call next
    decision = llm.invoke([
        SystemMessage("Route to: metrics_agent, "
                      "logs_agent, remediation_agent, or END"),
        HumanMessage(str(state["messages"]))
    ])
    return {"next": decision.content.strip()}

graph = StateGraph(AgentState)
graph.add_node("supervisor", supervisor)
graph.add_node("metrics",    metrics_agent)
graph.add_node("logs",       logs_agent)
graph.add_node("remediation",remediation_agent)

# Supervisor chooses the next node
graph.add_conditional_edges("supervisor", {
    "metrics_agent":     "metrics",
    "logs_agent":        "logs",
    "remediation_agent": "remediation",
    "END": END
})

💡 Each specialist agent runs its own ReAct loop with its own tools. The supervisor only decides who to call next — it never calls tools directly. This separation keeps each agent focused and debuggable.

from langchain_community.document_loaders import DirectoryLoader from langchain.text_splitter import RecursiveCharacterTextSplitter # Load all .md and .pdf runbooks from a folder loader = DirectoryLoader("./runbooks/", glob="**/*.md") docs = loader.load() # → 47 documents loaded # Split into overlapping chunks (1000 chars, 200 overlap) # Why overlap? So context isn't lost at chunk boundaries splitter = RecursiveCharacterTextSplitter( chunk_size=1000, chunk_overlap=200 ) chunks = splitter.split_documents(docs) # → 47 docs become 312 chunks
2
Embed & store in a vector database
from langchain_openai import OpenAIEmbeddings from langchain_chroma import Chroma # OpenAI converts each chunk to a 1536-dimension vector # "CrashLoopBackOff" → [0.023, -0.14, 0.87, ... 1536 numbers] embeddings = OpenAIEmbeddings(model="text-embedding-3-small") # Store all 312 chunk vectors in ChromaDB (local file) vectorstore = Chroma.from_documents( documents=chunks, embedding=embeddings, persist_directory="./chroma_db" ) # ← This runs ONCE. Next time: Chroma.load(persist_directory=...)
3
Query — retrieve relevant chunks + generate answer
from langchain.chains import RetrievalQA from langchain_openai import ChatOpenAI retriever = vectorstore.as_retriever( search_type="mmr", # MMR = diverse results, not just top-K similar search_kwargs={"k": 4} # fetch 4 most relevant chunks ) qa_chain = RetrievalQA.from_chain_type( llm=ChatOpenAI(model="gpt-4o-mini"), retriever=retriever, return_source_documents=True # show which runbooks were used ) result = qa_chain.invoke("How do I fix a CrashLoopBackOff pod?") print(result["result"]) # → "Based on runbook #3: First check logs with kubectl logs..." # → Sources: ['runbooks/k8s-troubleshooting.md', 'runbooks/pod-errors.md']
⚙️ Chunking strategy matters
  • Too small (200 chars) — chunks lose context, answers are fragmented
  • Too large (5000 chars) — retrieval is imprecise, irrelevant content included
  • Sweet spot: 800–1200 chars with 15–20% overlap
  • For structured docs (runbooks): split by heading (MarkdownHeaderSplitter)
💰 Embedding cost
  • text-embedding-3-small: $0.02 per 1M tokens
  • 312 chunks × 250 tokens avg = 78,000 tokens = $0.0016 to index
  • Indexing is one-time — queries are just retrieval (fast, cheap)
  • Queries cost only the LLM call (~$0.001–0.003 each)

👶 ELI5

Imagine a Jenkinsfile / GitHub Actions workflow — but each step is an AI agent that can think and use tools. LangGraph lets you draw that flowchart: define what each step does, how they connect, and what conditions decide the next step. It carries a shared state object (like a ticket) between every step, handles loops, manages checkpoints, and can pause to ask a human for approval. You write the logic; LangGraph handles all the execution plumbing.

🔧 Perfect DevOps Analogy
GitHub Actions
on: [push] jobs: test: { runs-on: ubuntu-latest } deploy: needs: test if: github.ref == 'main'
LangGraph
graph.add_node("analyze", analyze_fn) graph.add_node("remediate", fix_fn) graph.add_conditional_edges( "analyze", route_by_error_type)

Both define steps (nodes), dependencies (edges), and conditions (if branches). LangGraph just has AI at each step.

📐 Part 1: The Four Pillars

🧱 State — The Shared Memory of Your Workflow

State is a Python dict (TypedDict) that flows through every node. Each node reads it and returns updates. Think of it as an incident ticket being passed desk to desk.

PythonDefining State
from typing import TypedDict, Annotated, List
from langgraph.graph.message import add_messages

class IncidentState(TypedDict):
    # Reducer: new messages APPEND, not overwrite
    messages: Annotated[List, add_messages]

    # No reducer: new value OVERWRITES old
    incident_id:  str
    severity:     str
    error_type:   str | None
    resolution:   str | None
    approved:     bool
    retry_count:  int
ConceptWhat is a Reducer?
# WITHOUT reducer — last write wins:
state["error_type"] = "OOM"   # overwrites
state["error_type"] = "Crash" # "OOM" is GONE

# WITH add_messages reducer — appended:
state["messages"] += [AIMessage("Found OOM")]
state["messages"] += [HumanMessage("Fix it")]
# Both messages preserved in list!

# Custom reducer example:
def merge_findings(old, new):
    return {**old, **new}  # merge dicts

findings: Annotated[dict, merge_findings]
💡 add_messages is LangGraph's most-used built-in reducer — it powers the entire conversation history in agent workflows.

⭕ Nodes — The Workers in Your Workflow

A node is any Python function that takes the current state and returns a partial state update. Can be an LLM call, tool execution, plain code, or even another graph.

from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage, SystemMessage

llm = ChatOpenAI(model="gpt-4o")
llm_with_tools = llm.bind_tools([query_prometheus, run_kubectl, search_runbooks])

# Node type 1: LLM node — calls an LLM
def agent_node(state: IncidentState):
    system = SystemMessage("You are a DevOps SRE expert. Use tools to diagnose and fix issues.")
    response = llm_with_tools.invoke([system] + state["messages"])
    return {"messages": [response]}  # add_messages appends this

# Node type 2: Tool node — executes tool calls from LLM
from langgraph.prebuilt import ToolNode
tool_node = ToolNode([query_prometheus, run_kubectl, search_runbooks])
# ToolNode auto-reads tool_calls from last message, runs them, appends results

# Node type 3: Logic node — pure Python, no LLM
def severity_classifier(state: IncidentState):
    if state["error_type"] in ["OOMKilled", "CrashLoopBackOff"]:
        return {"severity": "P1"}
    return {"severity": "P2"}

# Adding nodes to graph:
graph = StateGraph(IncidentState)
graph.add_node("agent",      agent_node)
graph.add_node("tools",      tool_node)
graph.add_node("classify",   severity_classifier)

→ Edges — The Control Flow of Your Workflow

Edges define what happens after each node runs. There are three types:

Direct Edge

Always goes from A to B. No conditions.

# After "tools", always go back to "agent"
graph.add_edge("tools", "agent")
graph.add_edge("classify", "remediate")

Conditional Edge

Routing function decides which node to visit next based on state.

def should_continue(state):
    last = state["messages"][-1]
    if last.tool_calls:
        return "tools"  # has tool calls
    return END         # done

graph.add_conditional_edges(
    "agent", should_continue,
    {"tools": "tools", END: END}
)

Command Object

Node returns a Command to both update state AND control routing in one step.

from langgraph.types import Command

def router_node(state):
    error = classify(state)
    return Command(
        update={"error_type": error},
        goto="handle_oom" if error=="OOM"

🤖 Part 2: Building a Full ReAct Agent in LangGraph

Complete ReAct Agent — Production-Ready Code

This is the complete pattern for a ReAct agent. Study it carefully — this is the foundation of all LangGraph agent work:

from typing import Annotated
from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage
from langchain_core.tools import tool
from langgraph.graph import StateGraph, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode

# ── 1. Define State ──────────────────────────────────────
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

# ── 2. Define Tools ──────────────────────────────────────
@tool
def query_prometheus(query: str) -> str:
    """Run a PromQL query to get metrics. Use for CPU, memory, latency, error rate."""
    return prometheus_client.query(query)

@tool
def run_kubectl(command: str) -> str:
    """Run kubectl commands. Prefix write operations with --dry-run for safety."""
    return subprocess.check_output(["kubectl"] + command.split()).decode()

tools = [query_prometheus, run_kubectl]

# ── 3. Create LLM with tools bound ───────────────────────
llm = ChatOpenAI(model="gpt-4o", temperature=0)
llm_with_tools = llm.bind_tools(tools)

# ── 4. Define Nodes ──────────────────────────────────────
def agent_node(state: AgentState):
    """The brain: calls LLM which may output tool_calls or final answer"""
    system = SystemMessage(
        "You are a DevOps SRE. Use tools to diagnose Kubernetes incidents. "
        "Be methodical: check logs, then metrics, then suggest fixes."
    )
    response = llm_with_tools.invoke([system] + state["messages"])
    return {"messages": [response]}

tool_node = ToolNode(tools)  # Handles all tool execution automatically

# ── 5. Routing logic ─────────────────────────────────────
def should_continue(state: AgentState) -> str:
    """If last LLM message has tool_calls → execute tools. Otherwise → done."""
    last_message = state["messages"][-1]
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tools"
    return END

# ── 6. Build & compile graph ──────────────────────────────
graph = StateGraph(AgentState)

graph.add_node("agent", agent_node)
graph.add_node("tools", tool_node)

graph.set_entry_point("agent")
graph.add_conditional_edges("agent", should_continue, {"tools": "tools", END: END})
graph.add_edge("tools", "agent")  # Always loop back after tool use

app = graph.compile()

# ── 7. Run it! ────────────────────────────────────────────
result = app.invoke({
    "messages": [HumanMessage("api-server pod is in CrashLoopBackOff, investigate")]
})

from langgraph.checkpoint.memory import MemorySaver    # dev
from langgraph.checkpoint.sqlite import SqliteSaver   # prod (SQLite)
from langgraph.checkpoint.postgres import PostgresSaver # prod (Postgres)

config = {"configurable": {"thread_id": "incident-2024-001"}}

# The agent remembers the entire previous conversation
app.invoke({"messages": [HumanMessage("What was the root cause you found?")]}, config)

# View full history of checkpoints
for checkpoint in app.get_state_history(config):
    print(f"Step {checkpoint.config['checkpoint_id']}: {checkpoint.values}")

# REWIND: replay from a specific checkpoint (time travel!)
old_config = {"configurable": {"thread_id": "incident-2024-001", "checkpoint_id": "cp-step-3"}}

👤 Part 4: Human-in-the-Loop with interrupt()

Pausing Workflows for Human Approval

LangGraph's interrupt() lets you pause execution mid-workflow and wait for a human decision — essential for production where autonomous agents should ask before making dangerous changes.

from langgraph.types import interrupt, Command

def approval_gate_node(state: IncidentState):
    """Pause here and ask the on-call engineer for approval"""
    # interrupt() PAUSES the workflow and returns this data to the caller
    human_decision = interrupt({
        "message":  "⚠️ About to apply this fix to PRODUCTION:",
        "proposed": state["resolution"],
        "impact":   "Restarts api-server deployment (30s downtime)",
        "options":  ["approve", "reject", "modify"]
    })
    # When the human resumes, human_decision contains their response
    if human_decision == "approve":
        return {"approved": True}
    else:
        return {"approved": False, "rejection_reason": human_decision}

# ── Caller side ──────────────────────────────────────────
config = {"configurable": {"thread_id": "fix-approval-42"}}

# Step 1: Run until interrupt — workflow PAUSES at approval_gate_node
result = app.invoke(initial_state, config)
print(result["__interrupt__"])  # Shows the approval request to the engineer

# Step 2: Engineer reviews, then RESUMES workflow
final = app.invoke(Command(resume="approve"), config)
# Workflow continues from where it left off!

👑 Part 5: Supervisor Multi-Agent Pattern

Supervisor Pattern — Orchestrating Specialist Agents

A Supervisor is an LLM that reads the task and decides which specialist agent to call next. Each specialist agent is itself a ReAct agent with its own tools. This is the most powerful multi-agent pattern.

🎯 Supervisor LLM Routes task to correct specialist. Synthesises results.
📊Metrics AgentPrometheus, Grafana, CloudWatch
📋Logs AgentElasticsearch, Loki, CloudWatch Logs
☸️K8s Agentkubectl, Helm, K8s API
📚RAG AgentRunbooks, Past incidents, ADRs
from langgraph.graph import StateGraph, START, END
from langgraph.prebuilt import create_react_agent
from typing import Literal

# ── 1. Create specialist agents ───────────────────────────
metrics_agent = create_react_agent(llm, [query_prometheus, query_grafana],
    state_modifier="You specialise in metrics analysis.")
logs_agent    = create_react_agent(llm, [search_elasticsearch, tail_loki_logs],
    state_modifier="You specialise in log analysis.")
k8s_agent     = create_react_agent(llm, [run_kubectl, get_helm_values],
    state_modifier="You specialise in Kubernetes operations.")

# ── 2. Supervisor routes between agents ───────────────────
def supervisor_node(state: SupervisorState) -> Command[Literal["metrics","logs","k8s","__end__"]]:
    supervisor_prompt = """You coordinate a DevOps team.
    Route to: 'metrics' for metric/performance issues,
              'logs' for error/exception investigation,
              'k8s' for pod/deployment/resource issues,
              'FINISH' when investigation is complete."""
    response = llm.invoke([SystemMessage(supervisor_prompt)] + state["messages"])
    route = parse_route(response)  # extract routing decision
    if route == "FINISH":
        return Command(goto=END)
    return Command(goto=route, update={"messages": [response]})

# ── 3. Build the supervisor graph ─────────────────────────
supervisor = StateGraph(SupervisorState)
supervisor.add_node("supervisor", supervisor_node)
supervisor.add_node("metrics",    metrics_agent)
supervisor.add_node("logs",       logs_agent)
supervisor.add_node("k8s",        k8s_agent)

supervisor.add_edge(START, "supervisor")
# Each specialist reports back to supervisor after completing
for agent in ["metrics", "logs", "k8s"]:
    supervisor.add_edge(agent, "supervisor")

app = supervisor.compile()
🧠 Part 6: Memory Systems

Short-Term vs Long-Term Memory

🕐 Short-Term Memory

Conversation history within a single thread. Powered by checkpointing.

  • ✅ Automatic with MemorySaver/SqliteSaver
  • ✅ Scoped to one thread_id
  • ✅ Persistent across API calls
  • ⚠️ Limited by context window size
# Same thread = shared memory
config = {"configurable": {"thread_id": "inc-001"}}
app.invoke({...}, config)  # call 1
app.invoke({...}, config)  # call 2 — remembers call 1
🗄️ Long-Term Memory

Knowledge that persists across different threads and sessions. Stored in a memory store.

  • ✅ Cross-thread, cross-session
  • ✅ Semantic search (vector similarity)
  • ✅ Agent learns from past incidents
  • ⚠️ Requires explicit save/retrieve
from langgraph.store.memory import InMemoryStore
store = InMemoryStore()
# Save: "past incident → resolution"
store.put(("incidents","oom"), "k8s-oom-2024",
          {"fix": "increase memory limit"})
# Retrieve in any future workflow
past = store.search(("incidents",), query="oom")

🎯 Key Takeaways

  • State = TypedDict with reducers. add_messages appends; default overwrites.
  • Nodes = Python functions. Return partial state updates. Can be LLM, ToolNode, or plain Python.
  • Edges = Control flow. Direct (always), Conditional (if/else), or Command (update + route together).
  • Checkpointing = Persistent state per thread. Enables conversation memory and fault recovery.
  • interrupt() = Pause workflow for human approval. Resume with Command(resume=...).
  • Supervisor = Best multi-agent pattern. Specialist agents with their own tools, orchestrated by an LLM router.
  • For DevOps: Build Supervisor with Metrics + Logs + K8s specialist agents + interrupt() for production changes

Pythoningest.py — load & chunk
from langchain_community.document_loaders import DirectoryLoader, PyPDFLoader, TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter

def load_documents(runbook_dir: str) -> list:
    all_docs = []
    # Load PDFs
    all_docs.extend(DirectoryLoader(runbook_dir, glob="**/*.pdf",
                    loader_cls=PyPDFLoader, show_progress=True).load())
    # Load Markdown
    all_docs.extend(DirectoryLoader(runbook_dir, glob="**/*.md",
                    loader_cls=TextLoader, show_progress=True).load())
    print(ff"Loaded {len(all_docs)} documents")
    return all_docs

def chunk_documents(docs: list) -> list:
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=1000,    # ~750 words per chunk
        chunk_overlap=200,  # overlap prevents losing context at boundaries
        separators=["\n\n", "\n", " ", ""]
    )
    chunks = splitter.split_documents(docs)
    print(ff"Created {len(chunks)} chunks")
    return chunks

Build the RAG Pipeline
Wire the retriever to the LLM with a system prompt that forces source citation and refuses to hallucinate outside the runbooks
Pythonrag_chain.py
from langchain_openai import ChatOpenAI
from langchain.chains import RetrievalQA
from langchain.prompts import PromptTemplate
from ingest import load_vector_store

PROMPT = PromptTemplate(
    input_variables=["context", "question"],
    template="""You are a DevOps assistant. Answer ONLY using the
provided runbook context. If the answer is not in the context,
say "I don't have a runbook for this — escalate to on-call."
Always cite the source document at the end.

Context: {context}
Question: {question}
Answer:"""
)

def build_rag_chain():
    vectordb  = load_vector_store()
    retriever = vectordb.as_retriever(
        search_type="mmr",          # avoids returning duplicate chunks
        search_kwargs={"k": 4}
    )
    llm   = ChatOpenAI(model="gpt-4o-mini", temperature=0)
    chain = RetrievalQA.from_chain_type(
        llm=llm, chain_type="stuff", retriever=retriever,
        return_source_documents=True,
        chain_type_kwargs={"prompt": PROMPT}
    )
    return chain

Define the kubectl Tools
Wrap real Kubernetes API calls as @tool-decorated functions so the LLM can call them during diagnosis. The LLM decides which to call based on the event type.
Pythonoperator/tools.py
from langchain_core.tools import tool
from k8s_client import get_k8s_clients

k8s = get_k8s_clients()

@tool
def kubectl_describe_pod(pod_name: str, namespace: str) -> str:
    """Get pod describe output — status, events, resource limits."""
    pod    = k8s["core"].read_namespaced_pod(pod_name, namespace)
    status = pod.status
    return ff"Phase:{status.phase} | Containers:{status.container_statuses}"

@tool
def get_pod_logs(pod_name: str, namespace: str, tail: int = 100) -> str:
    """Get last N lines of logs, including previous container if crashed."""
    try:
        return k8s["core"].read_namespaced_pod_log(
            pod_name, namespace, tail_lines=tail, previous=True
        ) or "No logs available"
    except Exception as e:
        return ff"Error: {e}"

@tool
def patch_deployment_memory(deploy_name: str, namespace: str, memory_limit: str) -> str:
    """Patch deployment memory limit. memory_limit e.g. '512Mi' or '1Gi'."""
    patch = {"spec": {"template": {"spec": {"containers": [
        {"name": deploy_name,
         "resources": {"limits": {"memory": memory_limit}}}
    ]}}}}
    k8s["apps"].patch_namespaced_deployment(deploy_name, namespace, patch)
    return ff"Patched {deploy_name}: memory → {memory_limit}"

ALL_TOOLS = [kubectl_describe_pod, get_pod_logs, patch_deployment_memory]

Build the LangGraph Nodes
Each node is a pure Python function. The diagnostician node uses the LLM bound to tools. The approval node uses interrupt() to pause the graph and wait for human input.
Pythonoperator/nodes.py
from langchain_openai import ChatOpenAI
from langgraph.prebuilt import ToolNode
from langgraph.types import interrupt
from langchain_core.messages import HumanMessage
from tools import ALL_TOOLS, patch_deployment_memory

llm = ChatOpenAI(model="gpt-4o", temperature=0).bind_tools(ALL_TOOLS)

def classify_node(state):
    actionable = {"OOMKilling", "BackOff", "Failed", "CrashLoopBackOff"}
    is_act = state["event_reason"] in actionable
    return {"is_actionable": is_act, "issue_type": state["event_reason"]}

def diagnose_node(state):
    prompt = HumanMessage(content=f"""
Pod {state['pod_name']} in {state['namespace']}: {state['issue_type']}.
Use kubectl_describe_pod and get_pod_logs to diagnose root cause.
Then propose a specific remediation patch as JSON.
""")
    response = llm.invoke(state["messages"] + [prompt])
    return {
        "messages"      : [prompt, response],
        "diagnosis"     : response.content,
        "proposed_patch": {"memory_limit": "512Mi"}   # parse from LLM in production
    }

def approval_node(state):
    # ⏸ Graph PAUSES here — state is checkpointed to Redis
    decision = interrupt({
        "message"       : "Apply the proposed patch?",
        "pod"           : state["pod_name"],
        "diagnosis"     : state["diagnosis"],
        "proposed_patch": state["proposed_patch"]
    })
    return {"human_approved": decision}     # True or False from Slack

def patch_node(state):
    if not state["human_approved"]:
        return {"patch_result": "Rejected — escalated to on-call"}
    result = patch_deployment_memory.invoke({
        "deploy_name"  : state["pod_name"].rsplit("-", 2)[0],
        "namespace"    : state["namespace"],
        "memory_limit" : state["proposed_patch"]["memory_limit"]
    })
    return {"patch_result": result}

tool_node = ToolNode(ALL_TOOLS)

Assemble the StateGraph
Wire nodes with conditional edges. interrupt_before=["approval"] tells LangGraph to pause, checkpoint to Redis, and return — the graph only resumes when you call Command(resume=)
Pythonoperator/graph.py
from langgraph.graph import StateGraph, END
from langgraph.checkpoint.redis import AsyncRedisSaver
from state import K8sOperatorState
from nodes import classify_node, diagnose_node, approval_node, patch_node, tool_node

def route_after_diagnose(state):
    last = state["messages"][-1]
    return "tools" if getattr(last, "tool_calls", None) else "approval"

def route_after_classify(state):
    return "diagnose" if state["is_actionable"] else END

builder = StateGraph(K8sOperatorState)
builder.add_node("classify", classify_node)
builder.add_node("diagnose", diagnose_node)
builder.add_node("tools"   , tool_node)
builder.add_node("approval", approval_node)
builder.add_node("patch"   , patch_node)

builder.set_entry_point("classify")
builder.add_conditional_edges("classify", route_after_classify)
builder.add_conditional_edges("diagnose", route_after_diagnose,
                              {"tools": "tools", "approval": "approval"})
builder.add_edge("tools",    "diagnose")   # loop: tool result → re-invoke LLM
builder.add_edge("approval", "patch")
builder.add_edge("patch",    END)

checkpointer = AsyncRedisSaver.from_conn_string("redis://redis:6379")
graph = builder.compile(
    checkpointer=checkpointer,
    interrupt_before=["approval"]   # ⏸ pause & wait for human
)

from kubernetes import watch
from graph import graph
import asyncio, uuid

async def watch_k8s_events():
    w   = watch.Watch()
    k8s = get_k8s_clients()
    async for event in w.stream(k8s["core"].list_event_for_all_namespaces):
        obj = event["object"]
        if obj.reason not in {"OOMKilling", "BackOff", "Failed"}: continue
        thread_id = ff"{obj.involved_object.name}-{uuid.uuid4().hex[:8]}"
        config    = {"configurable": {"thread_id": thread_id}}

            "namespace"   : obj.involved_object.namespace,
            "messages"    : []
        }
        # Runs until interrupt() — pauses, saves to Redis, returns
        await graph.ainvoke(initial, config)

Key Design Rules

🆚
schemas/ vs models/
schemas/ = data shape (TypedDicts, Pydantic models, LangGraph State). models/ = AI model config (LLM clients, embeddings init). Never mix — one is your API contract, the other is your infra layer.
⚙️ Like protobuf definitions vs gRPC client setup
🔀
graphs/ owns flow, agents/ owns behaviour
graphs/ defines the DAG — nodes, edges, conditions. agents/ is the reasoning unit inside each node. An agent should not know what graph it sits in.
⚙️ Like Kubernetes Deployment spec vs container logic
📋
prompts/ as first-class assets
Never hardcode prompts in Python files. Store as .txt or .jinja2, load via prompts/loader.py. Version-controllable, reviewable by non-engineers, swappable without a code deploy.
⚙️ Like ConfigMaps — config shouldn't be baked into the image
🛡️
guardrails/ stays outside agents
Input/output validation, PII checks, prompt injection detection, human-in-the-loop gates — all live here. Agents call guardrails; guardrails never call agents. Same reason firewall rules don't live inside your app code.
⚙️ Like a WAF in front of your service, not inside it
🌿
configs/ follows 12-Factor
base.yaml holds shared defaults. Each env (dev, staging, prod) overrides only what differs. Load via a config loader that deep-merges them. No secrets in YAML — those go in .env / vault.
⚙️ Like Helm values.yaml + environment overrides
🧪
tests/evals/ separate from unit tests
LLM evals are probabilistic — they don't pass or fail deterministically. Keep them in tests/evals/ and run on a separate CI step. Never let flaky model output break a standard pytest run.
⚙️ Like chaos tests — run separately, not on every PR
🚫
No circular imports
Allowed: agents/ → schemas/, tools/, prompts/. Never: schemas/ or tools/ importing from agents/ or graphs/. Draw the dependency arrow once — it only points inward.
⚙️ Like service mesh traffic rules — unidirectional only
🗑️
Don't let utils/ become a dumping ground
If utils/ grows past 3–4 files, it needs a real name. utils/k8s_helpers.py → tools/kubernetes/. utils/llm_cache.py → memory/. A vague folder name hides architecture debt.
⚙️ Like the misc runbook — if it's important, give it a real name

Module 05 · RAG RAG searches runbooks for fix procedures

Before proposing a fix, the agent queries the runbook vector database. It embeds the error message, finds the most similar runbook chunks, and gives the LLM context-specific fix instructions.

# tools/analysis/runbook_search.py @tool def search_runbooks(query: str) -> str: """Search internal runbooks for known fixes.""" # Query = "DATABASE_URL missing environment variable crash" results = vectorstore.similarity_search(query, k=3) return "\n---\n".join([r.page_content for r in results]) # RAG returns relevant runbook section: "Runbook: Missing Env Vars After Deployment Cause: ConfigMap or Secret not mounted in new deployment Fix Option A: kubectl set env deployment/<name> VAR=value Fix Option B: helm upgrade <release> --set env.DATABASE_URL=$URL Fix Option C: Rollback: helm rollback <release> to last known-good"
