Terminal Setup

Installing and configuring WezTerm on macOS (Apple Silicon), including the Nerd Font, the full wezterm.lua, and the troubleshooting steps that actually got it working.

01

Install the Hack Nerd Font

WezTerm's config references this font by name, so it needs to exist before the config will render correctly.

zsh
➜ brew install --cask font-hack-nerd-font ==> Installing Cask font-hack-nerd-font 🍺 font-hack-nerd-font was successfully installed!
02

Create the config folder

WezTerm looks for its config at ~/.config/wezterm/wezterm.lua on macOS.

zsh
➜ mkdir -p ~/.config/wezterm
03

Install WezTerm itself

Easy to skip since the font and config feel like "the setup" — the app still has to be installed separately.

! Don't skip this one. The font and config can be fully in place while the app itself is still missing — which wezterm will come back empty and nothing will look different.
zsh
➜ brew install --cask wezterm
04

Write the config file

Full contents of ~/.config/wezterm/wezterm.lua — OS-aware, with separate branches for macOS and Windows.

~/.config/wezterm/wezterm.lua
local wezterm = require("wezterm") local config = wezterm.config_builder() local is_windows = os.getenv("OS") and os.getenv("OS"):lower():find("windows") local is_macos = wezterm.target_triple:lower():find("darwin") ~= nil config.color_scheme = "rose-pine-moon" config.max_fps = 120 config.font = wezterm.font("Hack Nerd Font", { weight = "DemiBold" }) config.window_decorations = "INTEGRATED_BUTTONS|RESIZE" config.window_frame = { font = wezterm.font("Hack Nerd Font", { weight = "Bold" }), } config.inactive_pane_hsb = { saturation = 0.0, brightness = 0.5, } if is_windows then config.win32_system_backdrop = "Acrylic" config.window_background_opacity = 0.7 config.window_frame.font_size = 10.0 end if is_macos then config.window_background_opacity = 0.8 config.macos_window_background_blur = 50 config.font_size = 15.0 config.window_frame.font_size = 13.0 end return config

What each part actually does

SettingEffect
color_schemeBuilt-in Rose Pine Moon palette — no separate install needed.
max_fpsCaps rendering at 120fps for smoother scrolling on high refresh-rate displays.
fontHack Nerd Font, DemiBold weight, for the terminal body text.
window_decorationsMerges the close/minimize/zoom buttons into the tab bar instead of a separate title bar.
inactive_pane_hsbDesaturates and dims panes that aren't focused, so the active split stands out.
is_macos block0.8 window opacity, 50pt background blur, 15pt terminal font, 13pt frame font — macOS-only.
is_windows blockAcrylic backdrop, 0.7 opacity, 10pt frame font — only applies on Windows machines.
05

Launch WezTerm

Open it directly rather than assuming a terminal window is already WezTerm — Terminal.app and iTerm will happily run the same commands and show none of this styling.

zsh
➜ open -a WezTerm
i Config changes apply automatically on save. If a change doesn't seem to take, reload manually with Cmd + Shift + R inside the WezTerm window.

Troubleshooting checklist

In order — this is the exact sequence that found the actual problem last time (WezTerm was never installed).

zsh
➜ cat ~/.config/wezterm/wezterm.lua # confirms the file actually saved and has all 33 lines
zsh
➜ which wezterm ➜ ls /Applications/ | grep -i wezterm # confirms whether WezTerm is installed at all
zsh
➜ /Applications/WezTerm.app/Contents/MacOS/wezterm # runs WezTerm in the foreground so any config/startup error prints right here
zsh
➜ wezterm ls-fonts --config-file ~/.config/wezterm/wezterm.lua # validates the Lua file directly and surfaces syntax errors
✓ Root cause last time: the font and config were both correctly in place — WezTerm itself just hadn't been installed yet. brew install --cask wezterm fixed it.

01

Install tmux

Via Homebrew, same as WezTerm.

zsh

Create the config folder and file

tmux 3.1+ follows the XDG spec and looks for config at ~/.config/tmux/tmux.conf automatically — same pattern as WezTerm, no extra env vars needed.

zsh
➜ mkdir -p ~/.config/tmux && touch ~/.config/tmux/tmux.conf

Write the config

Ported setting-by-setting from the Home Manager module's typed options, plus its extraConfig block copied through as raw tmux syntax.

~/.config/tmux/tmux.conf

What each part does

SettingEffect
base-index / pane-base-indexWindows and panes start numbering at 1 instead of 0.
clock-mode-style 2424-hour clock in tmux's built-in clock view (prefix + t).
focus-events onLets apps inside tmux (like Neovim) know when a pane gains/loses focus.
mode-keys / status-keys viVi-style navigation in copy-mode and the command prompt.
mouse onClick to switch panes, drag to resize, scroll to scroll back.

pane-border-lines heavyThicker visual divider between panes.
h / j / k / l select-paneVim-style pane navigation, repeatable (-r) without re-pressing the prefix each time.
" and %Split the current pane vertically / horizontally, opening in the same working directory.
copy-pipe-and-cancel "pbcopy"Sends the selection straight to the macOS clipboard when copying in vi copy-mode.
04

Load it

Start a new session, or reload the config into a session that's already running.

zsh
➜ tmux # starts a fresh unnamed session using ~/.tmux.conf
inside tmux
➜ tmux source-file ~/.config/tmux/tmux.conf # reloads config without restarting the session
i Default prefix is still Ctrl + b — this config doesn't remap it, so pane splits are prefix then " or %, and pane navigation is prefix then h/j/k/l.
05

Start a named session

Always give sessions a name — makes it easy to find and re-attach later. The -A flag attaches if the session already exists, or creates it if it doesn't.

zsh
➜ tmux new-session -A -s devops-test # creates session named "devops-test", or attaches if it already exists
! Common typo: the command is tmux, not tumx. Easy to mix up — double-check before hitting Enter.
06

Detach and re-attach

Detaching leaves the session running in the background — everything keeps going. Re-attach from any new terminal window.

Detach (leave session running)

⌨ Ctrl + b, then release both keys, then press d
This is a two-step sequence — press the prefix first, let go, then the key. Not Ctrl+A+D (that's GNU Screen). Not Ctrl+D (that closes the shell entirely).

Check what sessions are running

zsh
➜ tmux ls devops-test: 1 windows (created Wed Jul 2 10:00:00 2025) # if the session appears here, it's alive — just detached

Re-attach

zsh
➜ tmux attach -t devops-test # or use new-session -A -s devops-test — same result
i If detaching closes your terminal window, don't worry — the session is still alive. Open a new terminal and run tmux ls to confirm, then re-attach.

Write the config — init.lua

Full contents of ~/.config/nvim/init.lua. Bootstraps lazy.nvim on first launch and registers the three plugins.

~/.config/nvim/init.lua
-- Bootstrap lazy.nvim local lazypath = vim.fn.stdpath("data") .. "/lazy/lazy.nvim" if not vim.loop.fs_stat(lazypath) then vim.fn.system({ "git", "clone", "--filter=blob:none", "https://github.com/folke/lazy.nvim.git", "--branch=stable", lazypath, }) end vim.opt.rtp:prepend(lazypath) -- Options vim.opt.number = true vim.opt.relativenumber = true vim.opt.tabstop = 2 vim.opt.shiftwidth = 2 vim.opt.expandtab = true vim.opt.termguicolors = true -- Leader key vim.g.mapleader = " " -- Plugins require("lazy").setup({ -- File navigation (replaces netrw) { "stevearc/oil.nvim", opts = {} }, -- Git: status, diffs, commits { "NeogitOrg/neogit", dependencies = { "nvim-lua/plenary.nvim", "sindrets/diffview.nvim", }, config = true, }, -- File finding and codebase grepping { "folke/snacks.nvim", priority = 1000, lazy = false, opts = { picker = { enabled = true } }, }, })

What each plugin does

PluginRole
lazy.nvimPlugin manager — bootstraps itself on first launch, then installs and loads everything else.
oil.nvimBrowse and edit the filesystem like a buffer. Replaces the built-in netrw file explorer.
neogitFull git UI inside Neovim — stage, commit, diff, push without leaving the editor.
snacks.nvimFuzzy file finder and codebase grep. Find any file or search across the whole project instantly.

Requires Apple Silicon and macOS 14 or later. The app ships as a cask — no CLI tool, just the app bundle.

02

Download a Whisper model

On first launch the app asks you to pick a model. Transcription runs entirely on-device using these models — nothing is sent to a server.

ModelDescriptionPick if
Parakeet v3Fastest processing and accurateBest default - start here
Parakeet v2Fastest processing, English-only, higher recallEnglish-only and want max speed

03

Grant microphone & accessibility permissions

Two permissions are required. macOS will prompt for both on first use — grant them from System Settings if the prompts don't appear.

PermissionWhy it's needed
MicrophoneTo capture your voice while recording
AccessibilityTo paste the transcription into whichever app is in focus
i System Settings → Privacy & Security → Accessibility → enable OpenSuperWhisper. Without this the text is transcribed but cannot be auto-pasted.
04

How to use it

Works in any app — terminal, browser, editor, chat. No mode switching or copy-paste needed.

Release Right ⌥ OptionWhisper transcribes locally, text is pasted at cursor

✓ Practical use: hold the shortcut, say a long prompt to Claude Code, release — the full text appears in the terminal. Faster than typing long agent instructions.

One-time setup on your machine. Do this once — every project you work on will inherit these settings automatically.

01

Install Claude Code

Via npm. Installs the claude CLI globally.

zsh
➜ npm install -g @anthropic-ai/claude-code
zsh

02

Agent instruction file structure

There are two levels of global instruction files. Both are read automatically — no config needed.

FileScope
~/AGENTS.mdUniversal — works across Claude Code, OpenCode, and any other agent harness
~/.claude/CLAUDE.mdClaude Code specific — supplements AGENTS.md with Claude-only preferences
./CLAUDE.mdProject — checked into git, shared with the team

03

Write ~/AGENTS.md

The primary global file. Placed at the home directory so every agent harness picks it up automatically — not just Claude Code.

~/AGENTS.md
# Manjesh's agent instructions These are common instructions for Manjesh's agents across all scenarios. ## Writing Style Write like an experienced engineer communicating with other engineers. - Never use the em dash "—". Use a plain dash "-" instead. - Avoid semicolons (`;`) to join multiple actions or ideas. - Don't compress multiple ideas into a single sentence. - Prefer complete, natural sentences over dense summaries. - Use commas, parentheses, or separate sentences instead. - Prefer natural, conversational technical writing over AI-sounding prose. - Avoid compressed lists inside a sentence such as: "processes the data (batch, sample, redact, enrich) and fans it out...". - Instead, explain actions naturally or use bullets when listing multiple items. - Keep sentences short and direct. ### Don't write A standalone service that sits between your apps and your backends. Apps ship OTLP to it; it processes the data (batch, sample, redact, enrich) and fans it out to one or more backends. ### Write instead The OpenTelemetry Collector sits between your applications and your observability backends. Applications send OTLP data to the collector. The collector can batch requests, sample telemetry, redact sensitive fields, enrich data with metadata, and then forward it to one or more backends. Because of this, applications don't need to know which backend they're sending data to. Routing is managed in one place. ### Tone Write as if you're: - Writing an internal engineering design document. - Explaining something in a pull request. - Responding to a teammate in Slack. - Documenting an architecture for future maintainers. Avoid sounding like a blog post or marketing material. ### Formatting - Use headings to organize longer responses. - Keep paragraphs under five lines where practical. - Use code blocks for commands, configuration, and examples. - Bold only important terms. - Avoid excessive emphasis. - When writing or substantially editing long Markdown files, put each full sentence on its own line. Preserve normal Markdown structure, but avoid wrapping multiple sentences onto one physical line. ### Goal Every response should read as though it was written by an experienced engineer, not generated by an AI. If there is a choice between sounding polished and sounding natural, always choose the more natural wording. ## General Guidelines - When writing commit messages, NEVER auto-add your agent name as co-author. - Never manually modify CHANGELOG.md files or any files that are marked as auto-generated. - When making technical decisions, do not give much weight to development cost. Instead, prefer quality, simplicity, robustness, scalability, and long term maintainability. ## Pull Requests and Branching Always create a branch off the default branch for any change. Never commit directly to the default branch. Name every branch as `<prefix>/<short-kebab-description>`, where `<prefix>` is one of the values in the table below. Keep each pull request focused on a single change. Every pull request description must include: - **Summary** - what changed. - **Reason** - why the change was needed. | Prefix | Use it for | Example | | --- | --- | --- | | `feature/*` | New features or enhancements | `feature/login`, `feature/payment-gateway` | | `hotfix/*` | Critical production fixes only | `hotfix/1.2.1-critical-login-issue` | | `chore/*` | Dependency updates, maintenance, refactoring, or other non-functional changes | `chore/update-aws-sdk` | | `security-fixes/*` | Security improvements and vulnerability fixes | `security-fixes/cognito-token-validation` | Use `hotfix/*` only for urgent fixes that must reach production quickly. For an ordinary, non-urgent bug fix, use `feature/*` when the fix is enhancement-sized, or `chore/*` when it is small maintenance. ## Bug Fixes and Testing - When doing bug fixes, always start with reproducing the bug in an E2E setting as closely aligned with how an end user would experience it. This makes sure you find the real problem so your fix will actually solve it. - When end-to-end testing a product, be picky about the UI you see and be obsessed with pixel perfection. If something clearly looks off, even if it is not directly related to what you are doing, try to get it fixed alongside the current task. - Apply that same high standard to engineering excellence: lint, test failures, and test flakiness. If you see one, even if it is not caused by what you are working on right now, still get it fixed. ## Planning and Discussion When asked to plan, discuss options, explain trade-offs, or present anything complex, use lavish as the first choice. Create an HTML artifact and open it with `lavish-axi <file>` so Manjesh can visually review and annotate it. Only skip lavish if Manjesh explicitly asks for a Markdown file or says "no lavish". ## Manjesh's Opinions When you are working on something that would benefit from being informed by Manjesh's viewpoints, read ~/OPINIONS.md to understand his perspective. ## Voice Profile When you are talking/posting on behalf of Manjesh using his identity, read ~/VOICE.md to see how Manjesh talks.

No Mistakes

A local git proxy that sits in front of your real remote. Push to no-mistakes instead of origin and it runs an AI-driven review-test-lint pipeline in a disposable worktree, then opens a clean PR only after every check passes. Nothing reaches the real remote until the gate is green.

01

Install no-mistakes

One-line installer. Puts the binary at ~/.local/bin/no-mistakes and symlinks it from ~/.no-mistakes/bin/.

zsh
➜ curl -fsSL https://raw.githubusercontent.com/kunchenguid/no-mistakes/main/docs/install.sh | sh Downloading no-mistakes v1.31.2 for darwin/arm64... no-mistakes v1.31.2 installed to /Users/you/.no-mistakes/bin/no-mistakes Command path: /Users/you/.local/bin/no-mistakes -> /Users/you/.no-mistakes/bin/no-mistakes

Verify the install:

zsh
➜ no-mistakes --version no-mistakes version v1.31.2
02

Initialize a repo (per project)

Run this once inside each repo you want to gate. It adds a no-mistakes git remote pointing to a local bare repo and installs the /no-mistakes skill for Claude Code at user level.

zsh
➜ no-mistakes init ✓ Gate initialized repo /Users/you/src/my-repo gate no-mistakes → /Users/you/.no-mistakes/repos/abc123def456.git remote git@github.com:you/my-repo.git skill /no-mistakes installed for agents at user level Push through the gate with: git push no-mistakes <branch>

For fork-based repos (push to your fork, PR targets upstream):

zsh
➜ no-mistakes init --fork-url git@github.com:manjesh-raj/repo-name.git
03

Three ways to trigger the gate

Command When to use
git push no-mistakes Already have a committed branch - push through the gate instead of origin
no-mistakes Opens the TUI wizard - confirms branch, commit, then push. Add -y to skip all prompts
/no-mistakes <task> Agent skill - tell Claude Code to do a task and gate it, or bare /no-mistakes to gate existing committed work

When the TUI appears, it shows the current branch and commit state, then asks to confirm the push. Use these keys:

TUI
┌ Setup ──────────────────────────────────────┐ - Branch already on deploy_engine_optimize_code - Commit no uncommitted changes ❙ Push push branch to no-mistakes gate? └─────────────────────────────────────────────┘ y push confirm — start the pipeline n skip cancel this step, continue setup q quit exit the TUI entirely

Press y to push through the gate and start the review pipeline.

04

How the pipeline works

Every push runs through the same pipeline in an isolated worktree. Your working tree is never touched while the pipeline is running.

pipeline
your branch │ git push no-mistakes ▼ ┌──────────────────────────────────────────────┐ │ disposable worktree — your work stays put │ │ review → test → docs → lint → push → PR → CI │ └──────────────────────────────────────────────┘ │ every check green ▼ clean PR, opened for you

Each step produces a finding. Safe mechanical fixes are applied automatically. Anything touching intent is escalated - approve, fix, or skip. Nothing reaches the real remote until every check is green.

Keeping Tools Updated

The setup panels install the kunchenguid crew — firstmate and its companion CLIs. This panel keeps them current. Every tool lives at github.com/kunchenguid/<name>, and none of these updates need sudo (the Homebrew node prefix and the binary install dirs are all user-writable). Each tool below shows how to check whether it is behind and the exact command to bring it current — update only what has actually fallen behind.

01

firstmate — git repo, self-update

The firstmate home is a git clone (origin is your personal fork manjesh-raj/firstmate). The preferred update is the built-in command, which fast-forwards this repo plus any secondmate homes and re-reads its instructions.

preferred
➜ /updatefirstmate # fast-forwards the firstmate home + secondmate homes, then re-reads instructions

If it has diverged (you have local commits, so a plain fast-forward is refused), merge from upstream instead:

diverged
➜ git -C <firstmate-home> remote add upstream https://github.com/kunchenguid/firstmate # once; ignore "already exists" ➜ git -C <firstmate-home> fetch upstream ➜ git -C <firstmate-home> merge --no-edit upstream/main

Check what's behind: compare local HEAD against upstream main.

check
➜ git -C <firstmate-home> log -1 # installed ➜ git ls-remote https://github.com/kunchenguid/firstmate main # latest
! Never run git clean. projects/, data/, config/, and state/ are gitignored, so a fetch/merge never touches your cloned projects or local fleet state — but git clean would delete exactly those ignored dirs. After updating, re-read AGENTS.md; the operating instructions may have changed. Optionally mirror to your fork with git -C <firstmate-home> push origin main (a clean fast-forward, no force needed).
02

axi CLIs — global npm packages

Four CLIs — gh-axi, chrome-devtools-axi, tasks-axi, and lavish-axi — are published to npm from the axi monorepo. Update them all in one command:

update
➜ npm install -g gh-axi@latest chrome-devtools-axi@latest tasks-axi@latest lavish-axi@latest

Check what's behind: compare the published version against your installed one for any package.

check
➜ npm view <pkg> version # latest published ➜ npm ls -g --depth=0 | grep axi # installed
i lavish-axi needs a restart to activate. The npm install does not disturb running review sessions, but the new version only goes live once the lavish server restarts: run lavish-axi stop (it relaunches on next use). That drops live review connections, so do it when you are not mid-review.
03

no-mistakes — self-contained binary

Installed as a binary in ~/.no-mistakes/bin, it ships its own updater. No npm, no reinstall script.

update
➜ no-mistakes update

Check what's behind: no-mistakes --version also prints an "update available" banner when it is behind.

check
➜ no-mistakes --version
! Resets the daemon. no-mistakes update also restarts the no-mistakes daemon — only run it when no validation pipeline is in flight, or you will kill a run in progress.
04

treehouse — release binary

Installed as a binary at /usr/local/bin/treehouse. It is not distributed via npm and has no self-updater — it is release-based, so you compare your version against the repo's latest release tag and reinstall the latest release binary when behind.

check
➜ treehouse --version # installed ➜ git ls-remote --tags --sort=-v:refname https://github.com/kunchenguid/treehouse | head -1 # latest release tag

When behind, install the latest release binary from the repo's releases page (or its install script).

05

gnhf — global npm CLI

Not installed by default. The same npm install -g both installs it fresh and updates an existing copy.

install / update
➜ npm install -g gnhf@latest

Check what's behind: same npm pattern as the axi CLIs.

check
➜ npm view gnhf version # latest ➜ npm ls -g --depth=0 | grep gnhf # installed
06

Safety checklist

Before running any of the above, keep these in mind:

  • firstmate: projects/, data/, config/, state/ are gitignored and must stay untouched — never git clean. Re-read AGENTS.md after updating.
  • lavish-axi: schedule the lavish-axi stop restart for a moment when no review is in progress — it drops live sessions.
  • no-mistakes: run no-mistakes update only when no validation is in flight; it resets the daemon.
  • No sudo: nothing here needs it. If a global install reports a permission error, stop and fix the npm prefix rather than forcing it.
  • Update only what is actually behind, and note the before/after version of anything you touch.

🎯 Key Takeaways

  • Every tool lives at github.com/kunchenguid/<name>; none need sudo.
  • npm CLIs (axi tools, gnhf): npm view <pkg> version to check, npm install -g <pkg>@latest to update.
  • Binaries: no-mistakes update self-updates; treehouse is reinstalled from its latest release tag.
  • firstmate: prefer /updatefirstmate; fall back to an upstream fetch + merge if diverged.
  • Mind the three side effects — the firstmate gitignore protection, the lavish restart timing, and the no-mistakes daemon reset.
