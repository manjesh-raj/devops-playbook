Helm Module 01

What is Helm?

The package manager for Kubernetes — install, upgrade, rollback in one command

👶 ELI5

Helm is the apt/yum for Kubernetes. Instead of running kubectl apply -f deploy.yaml, kubectl apply -f svc.yaml, kubectl apply -f ingress.yaml... (10+ files), you run helm install myapp ./mychart and Helm handles everything — versioned, rollbackable, configurable.

😤 Without Helm
kubectl apply -f deployment.yaml kubectl apply -f service.yaml kubectl apply -f ingress.yaml kubectl apply -f configmap.yaml kubectl apply -f hpa.yaml # 10+ files to manage manually # No version tracking # No easy rollback
⛵ With Helm
helm install myapp ./mychart \ --values prod.yaml # One command # Versioned release # helm rollback myapp 1
🔥 Real ScenarioPain First

The Problem Helm Solves — 3 Environments

Imagine you're a DevOps engineer deploying myapp to dev, staging, and production. You have 5 Kubernetes files:

😩 Without Helm — 15 files to manage
kubectl-manifests/ ├── dev/ │ ├── deployment.yaml ← replicas: 1, image: v1.3.0 │ ├── service.yaml ← port: 8080 │ ├── ingress.yaml ← dev.myapp.internal │ ├── configmap.yaml ← LOG_LEVEL=debug │ └── hpa.yaml ← disabled ├── staging/ │ ├── deployment.yaml ← replicas: 2, image: v1.3.0 │ ├── ... ← copied & tweaked manually │ └── hpa.yaml ← minReplicas: 2 └── production/ ├── deployment.yaml ← replicas: 5, image: v1.2.9 ├── ... └── hpa.yaml ← minReplicas: 5, max: 20 # Problem: image v1.3.0 has a bug. # You must edit 3 deployment.yamls manually. # Miss one → production still runs buggy version.
✅ With Helm — 1 chart + 3 values files
myapp/ ← ONE chart (templates) ├── Chart.yaml ├── templates/ │ ├── deployment.yaml ← image: {{ .Values.image.tag }} │ ├── service.yaml │ ├── ingress.yaml ← {{ if .Values.ingress.enabled }} │ ├── configmap.yaml │ └── hpa.yaml ← {{ if .Values.autoscaling.enabled }} ├── values.yaml ← base defaults ├── values-dev.yaml ← replicas:1, debug logs, no HPA ├── values-staging.yaml ← replicas:2, HPA min 2 └── values-prod.yaml ← replicas:5, HPA min 5, max 20 # Bug in v1.3.0? Fix in ONE place: values-prod.yaml # image: # tag: v1.2.9 ← rollback all 3 envs in seconds
helm install myapp ./myapp
Dev — uses values.yaml defaults
→
helm install myapp ./myapp -f values-staging.yaml
Staging — 2 replicas, HPA enabled
→
helm install myapp ./myapp -f values-prod.yaml
Production — 5 replicas, TLS, alerts

💡 This is why Helm exists. One chart, many environments. Change the image tag in one values file and deploy everywhere. Zero copy-paste drift.

Chart Metadata — The Passport
Every chart must have a Chart.yaml. It tells Helm the chart name, version, and what app version it deploys. The version field must follow semver — this is what gets bumped on every release.
Example
apiVersion: v2
name: mychart
description: A production-ready app chart
type: application
version: 1.3.0       # Chart version (semver)
appVersion: "2.4.1" # App image tag
maintainers:
  - name: platform-team
    email: platform@company.io
💡 version = chart package version (bump when chart logic changes). appVersion = the app you're deploying (informational only — doesn't affect Helm logic).

⚙️ values.yaml — The Complete Guide

The single most important file in your chart — everything you need to know

👶 ELI5

values.yaml is your chart's configuration knob panel. Every setting that might need to change — image tag, replica count, memory limits, domain name, feature flags — goes here instead of hardcoded in your templates. When you deploy to dev, you leave it as-is. When you deploy to prod, you override just the values that differ. One chart, infinite configurations.

😩 Without values.yaml
# deployment.yaml — hardcoded everything image: myapp:v1.3.0 # ← change for every deploy replicas: 5 # ← different per env memory: "512Mi" # ← different per env host: myapp.prod.com # ← different per env # To deploy dev: manually edit 4+ values # To deploy staging: manually edit again # To update image tag: touch every env's file
✅ With values.yaml
# deployment.yaml — parameterised image: myapp:{{ .Values.image.tag }} replicas: {{ .Values.replicaCount }} memory: {{ .Values.resources.memory }} host: {{ .Values.ingress.host }} # To deploy dev: helm install myapp ./chart # To deploy prod: helm install myapp ./chart \ # -f values-prod.yaml # To update image: --set image.tag=v1.4.0

📄 Anatomy of a Well-Structured values.yaml

A production-grade values file has clear sections, helpful comments, and safe defaults

values.yamlFull annotated example
# ── Replica count ──────────────────────────────────────────────
# Number of pod replicas. Override in values-prod.yaml for scale.
replicaCount: 1

# ── Container image ────────────────────────────────────────────
image:
  repository: myregistry.io/myapp  # full registry path
  tag: "v1.0.0"                     # override via --set image.tag=
  pullPolicy: IfNotPresent          # Always in prod, IfNotPresent in dev

# ── Service ────────────────────────────────────────────────────
service:
  type: ClusterIP   # ClusterIP | NodePort | LoadBalancer
  port: 80
  targetPort: 8080  # port your app actually listens on

# ── Ingress (off by default — enable per environment) ──────────
ingress:
  enabled: false          # set true in values-staging/prod.yaml
  host: ""               # myapp.example.com
  tlsEnabled: false
  annotations: {}        # nginx.ingress.kubernetes.io/... etc

# ── Resource limits (ALWAYS set these — no limits = cluster risk) ─
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "256Mi"

# ── Horizontal Pod Autoscaler (off by default) ─────────────────
autoscaling:
  enabled: false
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70

# ── Application config (non-secret env vars) ───────────────────
config:
  LOG_LEVEL: "info"        # override to "debug" in dev
  DB_HOST: "postgres.svc"
  CACHE_TTL: "300"

# ── Secrets (reference only — never put real values here!) ─────
# Use: helm install --set secrets.dbPassword=$DB_PASS
# Or:  helm-secrets + SOPS / External Secrets Operator
secrets:
  dbPassword: ""          # injected via CI --set, never committed
  apiKey: ""

🔢 Data Types in values.yaml — Know Your YAML

YAML auto-detects types. This catches beginners off-guard constantly.

TypeExampleGotchaSafe Pattern
String tag: v1.0.0 tag: 1.0 → parsed as float 1.0 Always quote: tag: "v1.0.0"
Integer port: 8080 Fine as-is — YAML parses unquoted numbers as int port: 8080 ✓
Boolean enabled: true enabled: "true" (string) ≠ true (bool) in Go templates No quotes for booleans: enabled: false
Null / empty annotations: {} annotations: (bare) = null, breaks toYaml pipeline Use {} for empty maps, [] for empty lists
Multi-word string name: my app Spaces make it ambiguous — YAML may mis-parse Quote it: name: "my app"
List hosts: [a.com, b.com] Inline list vs block list — both valid but be consistent Block form preferred for readability
# In templates: always pipe strings through | quote for safety name: {{ .Values.config.LOG_LEVEL | quote }} # → "info" # Check if a value is truly set (not empty string, not null) {{- if and .Values.ingress.host (ne .Values.ingress.host "") }} host: {{ .Values.ingress.host }} {{- end }}

🔀 The enabled Pattern — Conditional Resources

The most common and powerful pattern in Helm charts — toggle entire resources on/off per environment

values.yamlDefault off
# Safe defaults — minimal resources
ingress:
  enabled: false  # no ingress in dev

autoscaling:
  enabled: false  # no HPA in dev

monitoring:
  enabled: false  # no ServiceMonitor in dev
values-prod.yamlOverride — enable in prod
# Only override what changes
ingress:
  enabled: true
  host: myapp.example.com

autoscaling:
  enabled: true
  minReplicas: 3

monitoring:
  enabled: true
# In templates/ingress.yaml — entire file is conditional {{- if .Values.ingress.enabled }} apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: {{ .Release.Name }} spec: rules: - host: {{ .Values.ingress.host }} {{- end }} # If ingress.enabled=false → this entire file renders as EMPTY # → K8s never sees an Ingress object # → No resource created, no cleanup needed
🔍 Values Validation

values.schema.json — Validate Before Deploying

Add a schema file and Helm will reject invalid values before touching your cluster — like a TypeScript type checker for your config.

values.schema.jsonEnforce rules on input
{
  "$schema": "http://json-schema.org/draft-07/schema",
  "type": "object",
  "required": ["replicaCount", "image"],
  "properties": {
    "replicaCount": {
      "type": "integer",
      "minimum": 1,
      "maximum": 50
    },
    "image": {
      "type": "object",
      "required": ["repository", "tag"],
      "properties": {
        "repository": { "type": "string" },
        "tag":        { "type": "string" },
        "pullPolicy": {
          "type": "string",
          "enum": ["Always", "IfNotPresent", "Never"]
        }
      }
    },
    "service": {
      "type": "object",
      "properties": {
        "port": { "type": "integer", "minimum": 1, "maximum": 65535 }
      }
    }
  }
}
❌ Schema rejects bad input
helm install myapp ./chart \ --set replicaCount=100 Error: values don't meet the specifications of the schema: - replicaCount: Must be ≤ 50 helm install myapp ./chart \ --set image.pullPolicy=latest Error: image.pullPolicy must be one of: Always, IfNotPresent, Never
✅ Why this matters
  • Catches misconfigured values before pods start
  • Required fields enforce contract with users
  • Enum rules prevent typos in critical fields
  • Range checks block runaway replica counts
  • Self-documenting — schema IS the spec

🌐 Global Values — Sharing Config Across Subcharts

When your chart has subcharts (dependencies), use .Values.global to share values without duplicating them in every sub-values file.

parent chart values.yamlDefine globals once
# Global values — accessible to ALL subcharts
global:
  imageRegistry: myregistry.io
  environment:   production
  domain:        example.com
  imagePullSecrets:
    - myregistrysecret

# Per-subchart overrides (optional)
postgresql:       # subchart name
  auth:
    database: myappdb

redis:            # subchart name
  architecture: standalone
subchart templates/deployment.yamlAccess globals
# Subcharts access global via .Values.global
image:
  # Use global registry if set, else local value
  repository: {{- if .Values.global.imageRegistry }}
    {{ .Values.global.imageRegistry }}/myapp
  {{- else }}
    {{ .Values.image.repository }}
  {{- end }}

env:
  - name: ENVIRONMENT
    value: {{ .Values.global.environment | quote }}

imagePullSecrets:
  {{- toYaml .Values.global.imagePullSecrets | nindent 2 }}
💡 global values are merged down into every subchart automatically. You set the registry once at the parent level — all 3 subcharts use it.
🚨 Security

What NEVER Goes in values.yaml (and What to Do Instead)

❌ Never put in values.yamlWhy✅ Use instead
Database passwords, API keys, tokens Git-committed plain text — exposed forever in history --set secrets.dbPass=$DB_PASS in CI, or helm-secrets + SOPS
TLS certificates / private keys Extremely sensitive, large binary data in YAML is fragile cert-manager + K8s TLS secrets
Cloud provider credentials (AWS keys, GCP SA JSON) One accidental commit = full cloud account exposure Workload Identity / IRSA / Pod Identity
SSH private keys As above — immediate security incident if leaked K8s Secrets + External Secrets Operator
# ✅ Safe pattern: placeholder in values.yaml, real value injected in CI # values.yaml: secrets: dbPassword: "" # intentionally empty — never committed with real value # CI pipeline (GitHub Actions / Jenkins / ArgoCD): helm upgrade myapp ./chart \ --set secrets.dbPassword=${{ secrets.DB_PASSWORD }} # The real password lives only in your CI secrets vault, never in git

🔍 Inspecting Active Values at Runtime

Always verify what values Helm is actually using — especially after complex overrides

# See what values are active for a running release helm get values myapp # user-supplied values only helm get values myapp --all # ALL values including defaults helm get values myapp --revision 3 # what was active at revision 3 helm get values myapp -n production # specify namespace # Preview what values would be used BEFORE deploying helm install myapp ./chart \ -f values-prod.yaml \ --set image.tag=v2.0.0 \ --dry-run --debug \ 2>&1 | grep -A 50 "COMPUTED VALUES" # see merged result # Render templates with specific values (no cluster needed) helm template myapp ./chart -f values-prod.yaml

🎯 Key Takeaways

  • values.yaml = safe dev defaults. Every env that needs different config gets its own values-{env}.yaml
  • Use the enabled: false pattern to conditionally include whole resources (ingress, HPA, monitoring)
  • Always quote string values that look like numbers or booleans. Use {} for empty maps, [] for empty lists
  • Add values.schema.json to validate input — catches misconfiguration before a single pod starts
  • Never commit secrets to values.yaml — use --set injection from CI secrets or External Secrets Operator
  • Use .Values.global in umbrella charts so all subcharts share registry, environment, and pull secrets from one place

Values Override Hierarchy — How Helm Merges Values

      
Helm merges values from multiple sources. The order of precedence (highest wins):


      

        

          
1 (highest)

          

            --set image.tag=v2.0.0
            CLI flag — overrides everything. Use for one-off tweaks or CI pipeline injection
          

        

        
▼ overrides

        

          
2

          

            -f values-prod.yaml
            Environment-specific values file passed with -f. Overrides defaults
          

        

        
▼ overrides

        

          
3 (lowest)

          

            values.yaml
            Chart defaults — always present, lowest priority. Everything starts here
          

        

      


      

        

          
values.yamlBase defaults

image:
  tag: "latest"
resources:
  memory: "128Mi"

  enabled: false

        

        

          
values-prod.yamlEnv override

          
# Only override what differs from base

  memory: "512Mi"

  enabled: true
  maxReplicas: 20

        

      


      
# Prod deploy: base values + prod override + CLI tag injection

  -f values-prod.yaml \
  --set image.tag=v2.1.0 \
  --namespace production

# replicaCount: 5         (from values-prod.yaml)
# image.tag: "v2.1.0"     (from --set, overrides prod file)
# resources.memory: 512Mi (from values-prod.yaml)
# autoscaling.enabled: true (from values-prod.yaml)

🔑 Zero-Downtime Secret

      
What Happens to Your Pods During helm upgrade?

      
This is the question every beginner has. Here's exactly what happens — step by step.


      

        

          
1

          

            You run helm upgrade myapp ./myapp --set image.tag=v2.0.0
            
Helm renders new templates, compares to current cluster state, applies the diff.

          

        

        

          
2

          

            K8s starts a Rolling Update (default Deployment strategy)
            
New pods with the new image are created first. Old pods are NOT immediately killed.

            
# Watch what happens:
kubectl get pods -w
myapp-v1-abc123   Running   ← still serving traffic
myapp-v2-def456   Pending   ← new pod starting up
myapp-v2-def456   Running   ← health checks pass
myapp-v1-abc123   Terminating ← NOW old pod is killed

          

        

        

          
3

          

            Readiness gates ensure zero downtime
            
K8s won't kill the old pod until the new one is Ready (passes readinessProbe). Traffic continues flowing to v1 while v2 starts. No requests are dropped.

          

        

        

          
4

          

            If the new pod fails health checks → Helm detects failure
            
With --atomic flag: Helm auto-rollbacks. Without it: release state = failed, old pods keep running until you fix it manually.

          

        

      


      

        

          
✅ Recommended: --atomic

          
helm upgrade myapp ./myapp \
  --set image.tag=v2.0.0 \
  --atomic \
  --timeout 5m

# If v2 pods don't become Ready in 5 min
# → Helm auto-rollbacks to v1
# → No manual intervention needed
# → Great for CI/CD pipelines

        

        

          
⚠️ Without --atomic

          
helm upgrade myapp ./myapp \
  --set image.tag=v2.0.0

# If upgrade fails:
# Release state = "failed"
# Your old pods are still running (good)
# But next helm upgrade will fail:
# Error: UPGRADE FAILED: another operation
# (install/upgrade/rollback) is in progress
# Fix: helm rollback myapp 2

🕰️ Version Control & Rollbacks

      
Every release is versioned — list history, inspect any revision, roll back in one command

    


    

      
👶 ELI5

      
Helm automatically keeps a full history of every deploy — like git log for your Kubernetes deployments. Each helm install or helm upgrade creates a new revision number. If something breaks, helm rollback myapp 2 instantly restores the exact state of revision 2 — the same templates, the same values, everything.

    


    
    

      
📦 How Helm Stores Release History

      
No external database. Helm stores each revision as a Kubernetes Secret in the same namespace.

      

        

          
# See the raw secrets Helm creates:
kubectl get secrets -n production \
  -l owner=helm,name=myapp

NAME                          TYPE
sh.helm.release.v1.myapp.v1   helm.sh/release.v1
sh.helm.release.v1.myapp.v2   helm.sh/release.v1
sh.helm.release.v1.myapp.v3   helm.sh/release.v1
sh.helm.release.v1.myapp.v4   helm.sh/release.v1

        

        

          
🏗️ What each secret contains

          
    
            
  • Rendered manifests — the exact YAML that was applied to the cluster at that revision
    • 
            
  • Computed values — all merged values (base + overrides + --set flags)
    • 
            
  • Chart metadata — chart name, version, app version
    • 
            
  • Release status — deployed / failed / superseded
    • 
            
  • Timestamps & description — when deployed and any description you set
    • 
          

          
💡 This is why helm rollback is so reliable — it replays the exact stored state, not a guess.

Helm Module 06

      
Hooks

      
Run jobs at specific lifecycle points — database migrations, smoke tests, notifications

    


    

      
👶 ELI5

      
Hooks are like CI/CD pipeline stages. A pre-install hook runs before Helm creates any resources — perfect for a DB migration job. A post-install hook runs after everything is up — perfect for a smoke test. Helm waits for each hook to complete before proceeding.

    


    

      
📋 Hook Reference

      

        

          
HookWhen it runsCommon use case

          
            
pre-installBefore any templates are appliedDatabase migration Job

            
post-installAfter all templates are runningSmoke test, send Slack notification

            
pre-upgradeBefore upgrade beginsBackup current state

            
post-upgradeAfter upgrade completesCache warm-up, verify health

            
pre-rollbackBefore rollback beginsNotify on-call, capture state

            
pre-deleteBefore uninstallData export, drain connections

            
testOn helm test commandConnection test, API health check

          
        

      

    


    

      
📝 Pre-install DB Migration Hook

      
YAMLtemplates/pre-install-migration.yaml

      
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ .Release.Name }}-db-migrate
  annotations:
    "helm.sh/hook": pre-install,pre-upgrade  # ← This is what makes it a hook
    "helm.sh/hook-weight": "-5"              # ← Lower = runs first
    "helm.sh/hook-delete-policy": hook-succeeded # ← Auto-cleanup
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
      - name: migrate
        image: myapp:{{ .Values.image.tag }}
        command: ["python", "manage.py", "migrate"]

      
💡 Hook weight: multiple hooks of the same type run in ascending weight order (-10 before -5 before 0 before 5). Use negative weights for hooks that must run earliest.

    


    

      
🗑️ Hook Deletion Policies

      

        
hook-succeeded
Delete the hook resource after it succeeds. Good for migrations — keeps the cluster clean.

        
hook-failed
Delete even if it fails. Use only when you have external logging — you'll lose the failure logs.

        
before-hook-creation
Default. Delete the old hook resource before running the new one. Allows retries on re-install.

        
(none)
Keep the resource forever. Useful for audit trail — but your cluster accumulates old Job objects.

Helm Module 08

      
Secrets & Security

      
The values.yaml problem, encryption strategies, and Helm 3 RBAC model

    


    

      
⚠️ The Problem

      
values.yaml is plaintext. If you put a database password in values.yaml and commit it, it's in your git history forever. Even worse — Helm stores release state as K8s Secrets, but those are only base64-encoded, not encrypted. Anyone with cluster access can read them.

    


    

      
🔐 Secrets Management Approaches

      

        

          
ApproachToolProsCons

          
            
Encrypted valueshelm-secrets + SOPSGit-safe, portableExtra tooling, key management

            
External secretsESO + Vault / AWS SMNo secrets in chartOperator required in cluster

            
Sealed SecretsBitnami Sealed SecretsK8s native, GitOps-friendlyCluster-specific (can't move certs)

            
CI env injection--set in pipelineSimple, no extra toolingSecrets appear in CI logs, no audit

          
        

      

    


    

      

        
🏆 Helm 3 Security Win

        
Tiller is Gone

        
Helm 2 had Tiller — a cluster-side daemon with cluster-admin privileges. It was a massive security hole. Helm 3 is client-only: it uses your kubeconfig credentials directly, so RBAC applies normally.

        
# Your CI ServiceAccount needs only:
- get/list/watch on Secrets
- create/update/delete on
  Deployments, Services, etc.
# NOT cluster-admin!

      

      

        
✅ Best Practices Checklist

        
    
          
  • ☑ Never commit real secrets to values.yaml
    • 
          
  • ☑ Use values.schema.json to validate all required fields
    • 
          
  • ☑ Pin dependency versions (avoid * or >=)
    • 
          
  • ☑ Use --atomic in CI to auto-rollback on failure
    • 
          
  • ☑ Verify chart integrity with helm verify
    • 
          
  • ☑ Least-privilege ServiceAccount for CI runners

👶 ELI5

      
Helm errors look scary but they follow patterns. Almost every error falls into one of 4 categories: template syntax (typo in {{ }}), values mismatch (wrong key names), cluster state (resource already exists), or app crash (your code broke, not Helm). Learn these patterns and you'll debug any issue in minutes.

              One chart = one application
              
All 5 resources belong to the same app — they go in the same chart. Helm manages them as a single unit: install all 5, upgrade all 5, rollback all 5 together.

            

          

          

            
❌

🚀 deployment.yaml — Before vs After

      
Replace hardcoded values with {{ .Values.xxx }} placeholders. Values come from values.yaml or --set on the command line.

      

        

          
❌ Before (kubectl)

          
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  namespace: production
  labels:
    app: myapp
    version: v2.1.0
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    spec:
      containers:
      - name: myapp
        image: registry.io/myapp:v2.1.0
        ports:
        - containerPort: 8080
        resources:
          requests:
            memory: "256Mi"
            cpu: "100m"
          limits:
            memory: "512Mi"
            cpu: "500m"

        

        
→

        

          
✅ After (Helm template)

          
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ .Release.Name }}
  namespace: {{ .Release.Namespace }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount }}
  selector:
    matchLabels:
      app: {{ .Release.Name }}
  template:
    spec:
      containers:
      - name: {{ .Chart.Name }}
        image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
        ports:
        - containerPort: {{ .Values.service.targetPort }}
        resources:
          {{- toYaml .Values.resources | nindent 10 }}

        

      

      
💡 {{ .Release.Name }} automatically uses whatever name you give in helm install <name> — so you can deploy multiple instances in different namespaces without editing the YAML.

    


    
    

      
🌐 service.yaml — Before vs After

      
Port and service type are the main variables here. Type can be ClusterIP in staging, LoadBalancer in prod — controlled by values.

      

        

          
❌ Before

          
apiVersion: v1
kind: Service
metadata:
  name: myapp-svc
spec:
  type: ClusterIP
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
    protocol: TCP

        

        
→

        

          
✅ After

          
apiVersion: v1
kind: Service
metadata:
  name: {{ .Release.Name }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  type: {{ .Values.service.type }}
  selector:
    app: {{ .Release.Name }}
  ports:
  - port: {{ .Values.service.port }}
    targetPort: {{ .Values.service.targetPort }}
    protocol: TCP

        

      

      
💡 Naming consistency: Use {{ .Release.Name }} for the Service name — the Deployment's selector also uses it, so they always match regardless of what name you pass to helm install.

    


    
    

      
🔀 ingress.yaml — Conditional + Parameterised

      
Ingress is optional — not every environment needs it. Wrap the whole file in {{- if .Values.ingress.enabled }} so it's only created when explicitly enabled.

      

        

          
❌ Before

          
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
  annotations:
    nginx.ingress.../rewrite-target: /
spec:
  rules:
  - host: myapp.company.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: myapp-svc
            port:
              number: 80

        

        
→

        

          
✅ After

          
{{- if .Values.ingress.enabled -}}
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: {{ .Release.Name }}
  {{- with .Values.ingress.annotations }}
  annotations:
    {{- toYaml . | nindent 4 }}
  {{- end }}
spec:
  rules:
  - host: {{ .Values.ingress.host }}
    http:
      paths:
      - path: {{ .Values.ingress.path | default "/" }}
        pathType: Prefix
        backend:
          service:
            name: {{ .Release.Name }}
            port:
              number: {{ .Values.service.port }}
{{- end }}

        

      

      
💡 Default disabled: Set ingress.enabled: false in values.yaml as the default. Prod overrides enable it. This way staging never accidentally exposes the app publicly.

    


    
    

      
⚙️ configmap.yaml — Environment Config

      
ConfigMap data is just key-value config. Dump the entire data: block from .Values.config using toYaml — one line replaces many hardcoded entries.

      

        

          
❌ Before

          
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
data:
  APP_ENV: "production"
  LOG_LEVEL: "info"
  DB_HOST: "postgres.internal"
  DB_PORT: "5432"
  CACHE_TTL: "300"
  FEATURE_X: "true"

        

        
→

        

          
✅ After

          
apiVersion: v1
kind: ConfigMap
metadata:
  name: {{ .Release.Name }}-config
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
data:
  {{- toYaml .Values.config | nindent 2 }}

# In values.yaml:
# config:
#   APP_ENV: "production"
#   LOG_LEVEL: "info"
#   DB_HOST: "postgres.internal"
#   DB_PORT: "5432"
#   CACHE_TTL: "300"
#   FEATURE_X: "true"

        

      

      
💡 toYaml . | nindent 2 dumps the entire config map as YAML with 2-space indent. Add or remove config keys purely in values.yaml — the template never needs to change.

    


    
    

      
📈 hpa.yaml — Auto-scaling Config

      
HPA min/max replicas and CPU threshold differ per environment. Dev might be 1–2, prod 3–20. Wrap in a conditional — not every deployment needs autoscaling.

      

        

          
❌ Before

          
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

        

        
→

        

          
✅ After

          
{{- if .Values.autoscaling.enabled -}}
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: {{ .Release.Name }}-hpa
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: {{ .Release.Name }}
  minReplicas: {{ .Values.autoscaling.minReplicas }}
  maxReplicas: {{ .Values.autoscaling.maxReplicas }}
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: {{ .Values.autoscaling.targetCPU }}
{{- end }}

        

      

      
💡 HPA + replicaCount conflict: When HPA is enabled, set replicaCount: 1 in values.yaml. The HPA takes over scaling — having a fixed replicaCount and HPA fight each other causes flapping.

🔒 What About Secrets?

        
Your configmap.yaml likely had a DB password or API key. That cannot go in values.yaml. Here are your options:

        

          

            1. helm-secrets + SOPS
            
Encrypt a secrets.yaml file. Git-safe, decrypted at deploy time.

            
helm secrets install myapp ./myapp \
  -f values-prod.yaml \
  -f secrets.enc.yaml

          

          

            2. --set in CI pipeline
            
Inject at deploy time from your CI secrets store. Simple but appears in CI logs.

            
helm upgrade --install myapp ./myapp \
  --set config.DB_PASSWORD=$DB_PASS

          

          

            3. External Secrets Operator
            
K8s operator that pulls secrets from Vault/AWS SM and creates K8s Secrets automatically.

            
# Chart just has an ExternalSecret CR
# ESO syncs the actual values

          

        

      

    


    
    

      
🚀 Deploy From Anywhere

      
Package your chart, push it to a registry, install from anywhere with a single command.


      

        

          
01

          

            Render & verify locally (no cluster needed)
            
helm template myapp ./myapp \
  -f values-prod.yaml

helm lint ./myapp -f values-prod.yaml

          

        

        

          
02

          

            Install to a cluster
            
# Dev (all defaults, no ingress, no HPA)
helm install myapp ./myapp

# Prod (override with prod values)
helm install myapp ./myapp \
  -f values-prod.yaml \
  --namespace production \
  --create-namespace \
  --atomic             # auto-rollback on failure

          

        

        

          
03

          

            Package the chart into a .tgz
            
helm package ./myapp
# Creates: myapp-1.0.0.tgz

          

        

        

          
04

          

            Push to an OCI registry (install from anywhere)
            
# Push to AWS ECR / GitHub / GCR / Harbor
helm push myapp-1.0.0.tgz \
  oci://registry.example.com/helm-charts

# Install from anywhere — no local files needed
helm install myapp \
  oci://registry.example.com/helm-charts/myapp \
  --version 1.0.0 \
  -f values-prod.yaml

          

        

        

          
05

          

            Upgrade (bump image tag, zero-downtime)
            
helm upgrade myapp \
  oci://registry.example.com/helm-charts/myapp \
  --version 1.1.0 \
  -f values-prod.yaml \
  --set image.tag=v2.2.0 \
  --atomic

# Something went wrong?
helm rollback myapp 1   # back to revision 1
helm history myapp      # see all revisions

🔍 Debug & Verify — When Things Go Wrong

      
Every beginner hits errors. Here's your debugging playbook for this exact chart.


      

        
🚨 The Most Common Errors

        

          

            
Error MessageCauseFix

            
              

                Error: INSTALLATION FAILED: cannot re-use a name that is still in use
                Release name already exists
                helm upgrade myapp ./myapp or helm uninstall myapp first
              

              

                Error: template: mychart/templates/...: function not defined
                Typo in template function name
                Run helm lint ./myapp — it will show the exact line
              

              

                CrashLoopBackOff after install
                App crashed — bad config or missing env var
                kubectl logs <pod-name> -n <namespace>
              

              

                ImagePullBackOff
                Wrong image tag in values.yaml
                Check image.tag value — fix and helm upgrade
              

              

                Error: UPGRADE FAILED: another operation is in progress
                Previous install/upgrade got stuck in pending state
                helm rollback myapp 0 to clear the pending state
              

              

                Error: rendered manifests contain a resource that already exists
                A resource was created outside Helm (raw kubectl apply)
                --force flag or delete the resource and reinstall
              

            
          

        

      


      
🔬 Debug Workflow — Step by Step

      

        

          
D1

          

            Before installing — validate your chart and values
            
# Lint checks YAML syntax and chart structure
helm lint ./myapp
helm lint ./myapp -f values-prod.yaml   # test with prod values

# Dry-run renders templates WITHOUT connecting to cluster
helm template myapp ./myapp -f values-prod.yaml

# Full dry-run: renders AND validates against cluster API
helm install myapp ./myapp \
  -f values-prod.yaml \
  --dry-run --debug \
  --namespace production

          

        

        

          
D2

          

            After installing — verify resources were created
            
# Check release status
helm status myapp -n production

# See all K8s resources this release owns
helm get manifest myapp -n production

# List all releases across namespaces
helm list -A

# Check pods are running
kubectl get pods -n production -l app.kubernetes.io/instance=myapp

          

        

        

          
D3

          

            Pod not starting? Investigate the logs
            
# Get pod name
kubectl get pods -n production
# → myapp-7d8f9-abc12  0/1  CrashLoopBackOff

# Read the crash logs
kubectl logs myapp-7d8f9-abc12 -n production
kubectl logs myapp-7d8f9-abc12 -n production --previous  # last crash

# Describe pod to see events (pull errors, OOM, etc.)
kubectl describe pod myapp-7d8f9-abc12 -n production

          

        

        

          
D4

          

            Check exactly what values Helm used for this release
            
# See computed values (after merging all overrides)
helm get values myapp -n production
helm get values myapp -n production --all   # include defaults

# See the rendered manifests Helm actually applied
helm get manifest myapp -n production

          

        

        

          
D5

          

            Fix → upgrade → verify loop
            
# Fix values-prod.yaml or template, then:
helm upgrade myapp ./myapp \
  -f values-prod.yaml \
  --atomic \
  --timeout 3m \
  --namespace production

# Check revision history
helm history myapp -n production

# If upgrade failed, rollback to last working revision
helm rollback myapp -n production  # rolls back to previous rev

          

        

      


      

        
🎖️ SRE Debugging Golden Rule

        
Always debug in this order: 1) helm lint → 2) helm template --dry-run → 3) helm install --dry-run --debug → 4) kubectl logs → 5) kubectl describe. 90% of all Helm failures are caught in the first 3 steps before a single pod is created.
