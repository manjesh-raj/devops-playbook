Module 01

What is AWS ADOT?

From "something is broken" to "here's exactly why" — ADOT is the observability layer every production system needs.

🧠 ELI5 — The Black Box Problem

Imagine your production system is a black box. Users say it's slow. Your boss says fix it. But you have no idea WHAT is slow, WHERE the bottleneck is, or WHY it started happening after last night's deployment. ADOT installs sensors everywhere inside that box — on every function call, every database query, every HTTP request. Now you can see exactly which database query took 258ms, which user's request triggered it, and that it started happening at exactly 14:32 — right after your migration ran.

🌳 Two Projects, One Tool
OpenTelemetry (OTel)
The open standard — a CNCF project (same home as Kubernetes, Prometheus). Vendor-neutral. Defines how observability data is collected and transmitted. Any vendor can implement it.
✅ Free, open source, community-driven
AWS Distro for OTel (ADOT)
AWS's hardened, enterprise-supported distribution of OpenTelemetry. Pre-configured for AWS services (X-Ray, CloudWatch, AMP). Tested, signed, and supported by AWS. Like Amazon Linux is to Linux.
✅ AWS-integrated, enterprise supported, security patched
💡 DevOps Analogy: OpenTelemetry is like Kubernetes — the open standard. ADOT is like EKS — AWS's managed, supported version. You get all the features plus AWS integrations and enterprise support.
⚡ Before ADOT / After ADOT
❌ Before: Tool Sprawl

Separate tools for each signal. All with different configs, different agents, different dashboards. Every new service needs to be wired up 3 separate times.

✅ After: One Collector

One ADOT Collector handles all three signals. One config file. One deployment. Automatically routes to the right AWS backend.

📅 Timeline — How We Got Here
2016–2018
Observability chaos: every vendor had their own agent. OpenCensus (Google) and OpenTracing (community) were two competing standards. Teams had to pick sides.
2019
OpenCensus + OpenTracing merge to create OpenTelemetry. CNCF adopts it. One standard to rule them all — traces, metrics, and eventually logs.
2020
AWS announces AWS Distro for OpenTelemetry (ADOT) in preview. AWS starts contributing heavily to the OpenTelemetry project.
2021
ADOT goes GA. AWS adds native integrations with X-Ray, CloudWatch, and Amazon Managed Service for Prometheus. EKS add-on support.
2023+
ADOT becomes the recommended observability approach for AWS workloads. Full support for all three signals (metrics, traces, logs) via OTel 1.0 stable spec.

Core Concepts — The Three Signals

Everything in observability boils down to three types of data. Master these and you can understand any monitoring system.

🚗 ELI5 — Your Car's Instrument Cluster

Metrics are like your car's gauges — speed (requests/sec), fuel (memory %), temperature (CPU %). Numbers that change over time. Traces are like GPS history — the exact path your package took from warehouse → truck → your door, with timestamps. Logs are like the driver's logbook — "14:32: left warehouse. 15:10: traffic jam on I-95. 15:45: delivered." Together they give you the full picture of what your system is doing.

📊 The Three Signals
NUMBERS OVER TIME
📈 Metrics

What: Numerical measurements sampled at intervals. Tells you how much and how fast.

DevOps examples:

  • CPU: 78%, Memory: 4.2GB
  • Requests/sec: 1,250 r/s
  • P99 latency: 45ms
  • Error rate: 0.3%
  • Active DB connections: 42

⚡ Answer: "How bad is it right now?"

REQUEST JOURNEY
🔍 Traces

What: The complete journey of one request through all services. Tells you where time was spent.

DevOps examples:

  • User clicked "Diagnose" → API Gateway → FastAPI → LangGraph → Redis → Postgres → Response
  • Total: 342ms. Postgres: 258ms (slow!)
  • Shows parent-child span relationships

⚡ Answer: "Which step is causing the slowness?"

TIMESTAMPED EVENTS
📝 Logs

What: Timestamped text records of discrete events. Tells you what happened.

DevOps examples:

  • 14:32 ERROR DB connection timeout
  • 14:32 WARN Retrying query (attempt 2/3)
  • 14:33 INFO Deployment abc1234 started

⚡ Answer: "What exactly happened at that moment?"

🎯 Traces Deep Dive — Spans Explained
ELI5: A Trace is a relay race. The full race = the trace. Each runner's leg = a span. Every span knows who passed it the baton (parent span) and how long it ran. The trace ID is the race number that stays the same across all runners.

🌐 Context Propagation — How Traces Cross Service Boundaries

When FastAPI calls Redis or Postgres, how does the trace know it's still the same request? The answer is Context Propagation — the trace ID is passed as an HTTP header (W3C TraceContext standard).

# Header automatically injected by ADOT SDK:
traceparent: 00-abc123xyz456-span001-01
#             ^^ trace-id       ^^ span-id

# Receiving service reads the header:
# "I'm part of trace abc123xyz456"
# → creates a child span with parent=span001
DevOps Analogy: VIP Wristband 🎫
At a festival, your VIP wristband stays with you all day. Security at each stage scans it to know you belong there and logs your visit. The wristband is your trace ID — it travels with every request across every service.

What gets propagated automatically:
  • Trace ID (unique per user request)
  • Span ID (the "current" step)
  • Sampling decision (don't re-decide mid-trace)
  • Baggage (custom key-value pairs)

ADOT Architecture — The Collector Pipeline

The ADOT Collector is the central hub. Everything flows through it — receive data, process it, export to AWS backends.

🏭 ELI5 — The Logistics Hub

The ADOT Collector is like an Amazon fulfillment center. Packages arrive from many carriers (receivers). Inside, they're sorted, labeled, consolidated, and some are inspected (processors). Then they're shipped to specific destinations — X-Ray gets traces, CloudWatch gets metrics, S3 gets logs (exporters). The warehouse doesn't care what's in the packages — it just routes them.

🧩 Receivers, Processors, Exporters — The Three Component Types
📥 Receivers
Data comes IN
  • otlp — standard OpenTelemetry (gRPC/HTTP)
  • prometheus — scrapes /metrics endpoints
  • awsecscontainermetrics — auto-collects ECS task/container stats
  • awscontainerinsightreceiver — EKS node/pod metrics
  • filelog — tail log files
  • kafka — consume from Kafka topics
⚙️ Processors
Data gets TRANSFORMED
  • batch — buffer data before exporting (efficiency)
  • probabilistic_sampler — keep X% of traces (cost control)
  • filter — drop health-check spans, noisy metrics
  • resource — add/change attributes (env, team, region)
  • attributes — rename or delete span attributes
  • memory_limiter — prevent OOM in the collector
📤 Exporters
Data goes OUT
  • awsxray — sends traces to AWS X-Ray
  • awsemf — CloudWatch metrics (EMF format)
  • prometheusremotewrite — sends to AMP
  • awss3 — archive telemetry to S3
  • otlp — forward to any OTLP backend
  • logging — debug: print to stdout

# adot-collector-config.yaml
receivers:
  otlp:
    protocols:
      grpc: { endpoint: 0.0.0.0:4317 }  # Used by apps
      http: { endpoint: 0.0.0.0:4318 }  # Used by Lambda
  awscontainerinsightreceiver:           # Auto: EKS node+pod metrics
    collection_interval: 30s
  prometheus:                            # Scrape FastAPI /metrics
    config:
      scrape_configs:
        - job_name: fastapi-sre
          static_configs:
            - targets: ['localhost:8000']

processors:
  batch:
    timeout: 5s
    send_batch_size: 1024     # Buffer before exporting (reduces API calls)
  probabilistic_sampler:
    hash_seed: 42
    sampling_percentage: 10   # Keep 10% of traces in high-traffic (save cost)
  resource:
    attributes:               # Add metadata to ALL telemetry
      - key: environment  value: "production"         action: upsert
      - key: service.team value: "platform-sre"      action: upsert
  filter/drop-health:         # Remove noisy health-check traces
    traces:
      span:
        - 'attributes["http.target"] == "/health"'

exporters:
  awsxray:
    region: us-east-1         # Traces → AWS X-Ray
  awsemf:
    namespace: SREAutopilot
    region: us-east-1         # Metrics → CloudWatch in EMF format
    log_group_name: /aws/sre-autopilot/metrics
  prometheusremotewrite:
    endpoint: "https://aps-workspaces.us-east-1.amazonaws.com/workspaces/ws-xxx/api/v1/remote_write"
    auth: { authenticator: sigv4auth }  # Metrics → Amazon Managed Prometheus

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch, probabilistic_sampler, resource, filter/drop-health]
      exporters: [awsxray]
    metrics:
      receivers: [otlp, awscontainerinsightreceiver, prometheus]
      processors: [batch, resource]
      exporters: [awsemf, prometheusremotewrite]

What to Monitor — The Complete Checklist

From the RED method to custom business metrics — here's exactly what to instrument and why.

🔴 The RED Method — For Every Service

Coined by Tom Wilkie at Weave. These three metrics tell you if a service is healthy. If any of these is wrong, users are unhappy.

🔴 R — Rate
How many requests per second?
  • HTTP requests/sec per route
  • DB queries/sec
  • Cache operations/sec
rate(http_requests_total[5m])
🟢 E — Errors
How many requests are failing?
  • HTTP 5xx responses
  • Unhandled exceptions
  • DB timeouts and errors
rate(http_errors_total[5m])
  / rate(http_requests_total[5m])
🔴 D — Duration
How long do requests take?
  • P50 (median) latency
  • P95 latency (most users)
  • P99 latency (worst 1%)
histogram_quantile(0.99,
  rate(http_duration_bucket[5m]))
💡 SRE Alert Targets
Error Rate:
Alert if > 1% over 5 minutes
P99 Latency:
Alert if > 500ms over 5 minutes
Request Rate drop:
Alert if drops > 30% from baseline (traffic lost)

🔗 Log Correlation — Connect Logs to Traces

Add the trace_id to every log line. Then in CloudWatch or OpenSearch, you can jump from any log entry directly to the full distributed trace.

import logging
from opentelemetry import trace

class TraceIdFilter(logging.Filter):
    """Inject trace_id and span_id into every log record."""
    def filter(self, record):
        ctx = trace.get_current_span().get_span_context()
        record.trace_id = format(ctx.trace_id, '032x') if ctx.is_valid else 'no-trace'
        record.span_id  = format(ctx.span_id,  '016x') if ctx.is_valid else 'no-span'
        return True

# Configure structured JSON logging
logging.basicConfig(format='{"ts":"%(asctime)s","level":"%(levelname)s",'
                            '"msg":"%(message)s","trace_id":"%(trace_id)s",'
                            '"span_id":"%(span_id)s"}')
logging.getLogger().addFilter(TraceIdFilter())

# Now every log line includes the trace ID:
# {"ts":"14:32:05","level":"ERROR","msg":"DB timeout",
#  "trace_id":"abc123xyz456","span_id":"span007"}
# → Click trace_id in CloudWatch → opens X-Ray trace instantly

AWS Integrations — Where Your Data Goes

ADOT routes each signal type to the AWS service purpose-built for it. Here's what each service does and when to use it.

🗺️ Signal → AWS Service Routing
TRACES ──→ AWS X-Ray (distributed trace UI, service map, anomaly detection) METRICS ──→ Amazon CloudWatch (dashboards, alarms, Container Insights) METRICS ──→ Amazon Managed Prometheus (PromQL queries, long-term storage) METRICS ──→ Amazon Managed Grafana (beautiful dashboards over AMP + CloudWatch) LOGS ──→ Amazon CloudWatch Logs (full-text search, log insights) LOGS ──→ Amazon OpenSearch Service (complex queries, ML-based anomaly detection)
☁️ AWS Service Deep Dives
🔍 AWS X-Ray
Distributed tracing — the "GPS history" of your requests. Shows service maps, end-to-end latency breakdowns, and identifies exactly which database query caused the slowdown.
Use it for: "Find all requests that took >500ms and show me which service was slow."
Signal: Traces

"""ADOT telemetry setup for SRE Auto-Pilot."""
import os, logging
from opentelemetry import trace, metrics
from opentelemetry.instrumentation.fastapi    import FastAPIInstrumentor
from opentelemetry.instrumentation.redis      import RedisInstrumentor
from opentelemetry.instrumentation.sqlalchemy import SQLAlchemyInstrumentor
from opentelemetry.instrumentation.httpx      import HTTPXClientInstrumentor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter  import OTLPSpanExporter
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter
from opentelemetry.sdk.trace   import TracerProvider
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.trace.export   import BatchSpanProcessor
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader
from opentelemetry.sdk.resources import Resource

COLLECTOR = os.getenv("OTEL_EXPORTER_OTLP_ENDPOINT", "http://localhost:4317")

def setup_telemetry(app=None):
    """Call once at startup — instruments everything."""
    resource = Resource.create({
        "service.name":             "sre-autopilot",
        "service.version":          os.getenv("IMAGE_TAG", "dev"),
        "deployment.environment":   os.getenv("ENV", "dev"),
        "k8s.namespace.name":       os.getenv("POD_NAMESPACE", "default"),
        "k8s.pod.name":             os.getenv("POD_NAME", "local"),
    })

    # ── Traces ──────────────────────────────────────────────
    tracer_provider = TracerProvider(resource=resource)
    tracer_provider.add_span_processor(
        BatchSpanProcessor(OTLPSpanExporter(endpoint=COLLECTOR, insecure=True))
    )
    trace.set_tracer_provider(tracer_provider)

    # ── Metrics ─────────────────────────────────────────────
    reader = PeriodicExportingMetricReader(
        OTLPMetricExporter(endpoint=COLLECTOR, insecure=True),
        export_interval_millis=15000   # every 15 seconds
    )
    meter_provider = MeterProvider(resource=resource, metric_readers=[reader])
    metrics.set_meter_provider(meter_provider)

    # ── Auto-instrument libraries ────────────────────────────
    if app: FastAPIInstrumentor.instrument_app(app)
    RedisInstrumentor().instrument()
    SQLAlchemyInstrumentor().instrument()
    HTTPXClientInstrumentor().instrument()

    return trace.get_tracer("sre-autopilot")

# telemetry.py
import os
from opentelemetry import trace, metrics
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.sdk.metrics import MeterProvider
from opentelemetry.sdk.metrics.export import PeriodicExportingMetricReader
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.exporter.otlp.proto.grpc.metric_exporter import OTLPMetricExporter
from opentelemetry.sdk.resources import Resource, SERVICE_NAME, SERVICE_VERSION
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.sqlalchemy import SQLAlchemyInstrumentor
from opentelemetry.instrumentation.redis import RedisInstrumentor
from opentelemetry.instrumentation.requests import RequestsInstrumentor
from opentelemetry.propagate import set_global_textmap
from opentelemetry.propagators.aws import AwsXRayPropagator
from opentelemetry.sdk.extension.aws.trace import AwsXRayIdGenerator

def setup_telemetry(app, db_engine=None):
    # OTLP endpoint = the ADOT Collector Service we deployed in Phase 4
    # adot-col-collector.adot-system.svc.cluster.local is the DNS name
    # the operator created automatically
    otlp_endpoint = os.getenv(
        "OTLP_ENDPOINT",
        "http://adot-col-collector.adot-system.svc.cluster.local:4317"
    )

    resource = Resource(attributes={
        SERVICE_NAME:    os.getenv("SERVICE_NAME", "my-app"),
        SERVICE_VERSION: os.getenv("SERVICE_VERSION", "1.0.0"),
        "deployment.environment": os.getenv("ENVIRONMENT", "production"),
        "k8s.cluster.name": os.getenv("CLUSTER_NAME", "unknown"),
    })

    # ── Traces ──────────────────────────────────────────
    # AwsXRayIdGenerator: formats trace IDs as X-Ray requires (1-{epoch}-{random})
    tracer_provider = TracerProvider(
        resource=resource,
        id_generator=AwsXRayIdGenerator()
    )
    tracer_provider.add_span_processor(
        BatchSpanProcessor(
            OTLPSpanExporter(endpoint=otlp_endpoint),
            max_queue_size=2048,
            max_export_batch_size=512,
            export_timeout_millis=30000,
        )
    )
    trace.set_tracer_provider(tracer_provider)

    # AwsXRayPropagator: reads/writes X-Amzn-Trace-Id header
    # This makes traces chain across service boundaries automatically
    set_global_textmap(AwsXRayPropagator())

    # ── Metrics ──────────────────────────────────────────
    meter_provider = MeterProvider(
        resource=resource,
        metric_readers=[
            PeriodicExportingMetricReader(
                OTLPMetricExporter(endpoint=otlp_endpoint),
                export_interval_millis=30_000,
            )
        ],
    )
    metrics.set_meter_provider(meter_provider)

    # ── Auto-instrumentation ─────────────────────────────
    FastAPIInstrumentor.instrument_app(app)  # all HTTP routes get spans
    RequestsInstrumentor().instrument()       # outbound HTTP calls get spans
    if db_engine:
        SQLAlchemyInstrumentor().instrument(engine=db_engine)

    RedisInstrumentor().instrument()

    return trace.get_tracer(os.getenv("SERVICE_NAME", "my-app"))

@app.get("/api/diagnose")

        # Attributes show up as X-Ray annotations — searchable in console
        span.set_attribute("sre.issue_type", issue)

  "ReservoirSize": 10,
  "FixedRate": 1.0,
  "URLPath": "*",
  "Host": "*",
  "HTTPMethod": "*",
  "ServiceName": "my-app",
  "ServiceType": "AWS::EKS::Container",
  "ResourceARN": "*",
  "Attributes": {}
}'

# Add to your app temporarily:
@app.get("/api/test/slow")
async def slow_endpoint():
    with tracer.start_as_current_span("slow.db.query") as span:
        span.set_attribute("db.statement", "SELECT * FROM events LIMIT 100000")
        span.set_attribute("db.system", "postgresql")
        await asyncio.sleep(2.3)   # simulate slow query
    return {"status": "done"}

# 1. Traces → filter: responsetime > 2
# 2. Click the 2.3s trace
# 3. Expand waterfall → see "slow.db.query" span = 2300ms
# 4. Click that span → see db.statement attribute with the slow SQL
# 5. This is how you find slow queries in production in 30 seconds
