1. What This Is

python_raasng_devops is a home-grown Python automation framework. It runs DevOps operations against remote hosts. It is the engine behind tenant onboarding, deboarding, database work, and a large catalogue of one-off infrastructure tasks.

The framework is written in Python and built on Fabric 1.x, which needs Python 2.7. Jenkins jobs drive it. Each job runs a small command-line entry point that does one task.

In Simple Words You describe a task once as a Python class. Jenkins runs it. Fabric carries the commands over SSH to the target host.

2. The Problems It Solves

Without a framework, every automation job is its own script. Each script re-invents SSH connection handling, secrets, logging, dry-run, and exit codes. That is slow to write and easy to get wrong.

This framework gives every task the same shape. Connection identity lives in one place. Secrets live in another. The base class handles execution, logging, dry-run, error policy, and exit codes for you.

What You Get For Free
  • One SSH connection layer for every host
  • Secrets kept separate from connection details
  • A uniform task shape
  • Ordered, repeatable multi-command runs
  • Built-in dry-run preview
  • Exit codes Jenkins can gate on

3. The Shape Of The Codebase

The design is a small layered core plus many thin task modules. There are around 112 task modules under lib/ and 109 matching entry points under bin/.

  • lib/config.py - the config core. It turns INI files into validated Python dicts.
  • lib/system_config.py - the connection layer. It holds SSH identity per host.
  • lib/pramata_config.py - the secrets layer. It holds per-host application credentials.
  • lib/commander.py - the execution engine. It generates commands and runs them through Fabric.
  • lib/<task>.py - one thin module per task. It subclasses the engine and lists the commands to run.
  • bin/execute_<task>.py - the CLI entry point that Jenkins invokes.
The rest of this notebook walks each of these layers. Pages 5 and 6 cover setup. Page 7 covers how the engine works.

Why This Approach Works

These advantages are grounded in what the code actually does. They are the reason the repo can carry 100+ task modules cheaply.

1. One Place For Connection Identity SystemConfig holds the SSH user, sudo user, password, and port per host, with a [DEFAULT] fallback. Every task inherits the same connection behaviour. There is no per-script SSH boilerplate.
2. Connection Split From Secrets Transport config and application secrets are separate layers. A task states what secrets it needs, apart from how to reach the host. Swapping the secret set is a one-line change.
3. Uniform Task Shape A new task is a subclass plus a few small methods that return command descriptors. The base class already handles execution, logging, dry-run, error policy, and exit codes. The cost to add automation stays low.
4. Ordered, Repeatable Runs The driver runs command generators in a deterministic order. It aggregates return codes and applies the same error policy across every task. A run behaves the same way every time.
5. Dry-Run Safety Built In A print_only flag lets an operator preview exactly what would run before executing. This matters for destructive tenant and database operations.
6. Jenkins-Friendly Exit Codes Each task has a bin/execute_*.py wrapper that parses options and calls run(). The engine maps aggregate failure to a process exit code, so a Jenkins job can gate pass or fail.
7. Typed, Explicit Errors Both layers raise named exceptions instead of leaking raw parser or IO errors. Config problems and command failures are easy to read in a Jenkins log.
8. Multi-Host Parallelism Available The engine can fan a command out across hosts with xargs -P. Multi-host config classes exist for when a task targets a fleet.

Setup, Part 1: The Jenkins Server

The framework runs from a Jenkins server. Jenkins triggers the tasks, and the tasks reach out to the target hosts over SSH. This page sets up that Jenkins runner.

1. Prerequisite

Start with a Jenkins server deployed from the AWS Marketplace. Use the Jenkins Certified by Bitnami image. Log in once with its credentials to confirm it is ready.

The devops User The framework runs as a dedicated devops user. Switch to it before setting anything up.

2. Switch To The devops User

Become the devops user on the Jenkins server 
sudo su - devops

3. Environment Variables And Aliases

Add these exports and aliases to the devops user's BASHRC. The PYTHONPATH entries are what let lib/ and Utils/lib/ import cleanly. Without them the config discovery fails.

Add to the devops user's BASHRC 
# base paths
export HOME="/opt"
export DEVOPS_BASE="$HOME/devops_tools/python_devops"
export PYTHONPATH="$PYTHONPATH:$DEVOPS_BASE/lib:$DEVOPS_BASE/Utils/lib:"

# handy aliases
alias deploy="cd $DEVOPS_BASE/lib"
alias eb="vi $BASHRC"
alias resource="source $BASHRC"
  • deploy jumps you into the lib/ directory where the task modules live.
  • eb opens the BASHRC for editing.
  • resource re-sources the BASHRC after a change.

4. Install Fabric

The framework needs Fabric 1.x on Python 2.7. Install it, then verify the version. You should see Fabric 1.x and Paramiko, not Fabric 2 or 3.

Install Fabric and check the version 
sudo apt-get install fabric

# verify: expect Fabric 1.10.2, Paramiko 1.16.0
fab -V

5. Clone The Code Onto The Server

As root, clone the framework into /opt, then hand it to the devops user. The chown step matters, since the tasks run as devops.

Clone as root, then chown to devops 
sudo su -
cd /opt
git clone https://github.com/manjesh-raj/python_raasng_devops.git
chown -R devops: python_raasng_devops
The deploy path the tasks expect is under /opt. Keep the code there so the DEVOPS_BASE and PYTHONPATH above line up.

Setup, Part 2: The Target Server

The Jenkins server reaches each target host over SSH as the devops user. So every target needs that user and the right SSH settings. Bake these changes into the AMI, so every new instance is ready out of the box.

Do this in the base image, not by hand on each server. A one-time AMI change means every target host that boots from it can already be reached.

1. Create The devops User

Create the linux devops user on the target 
adduser devops

Create the user in the image, then set the password on the running EC2 instance.

2. Add The Sudoers Entries

Give the devops user sudo rights. Add these lines to the sudoers config.

Sudoers entries 
%wheel  ALL=(ALL)  ALL
%devops ALL=(ALL)  ALL

3. Configure SSH

Edit /etc/ssh/sshd_config. Turn on password authentication and limit logins to the devops user.

Edit /etc/ssh/sshd_config 
# in /etc/ssh/sshd_config
PasswordAuthentication yes
AllowUsers devops

4. Restart SSH

Apply the SSH config change 
service sshd restart

5. Open Port 22

Open port 22 from the Jenkins server to the target server. This is what lets Fabric open the SSH session. Scope the rule to the Jenkins server, not the whole internet.

Checklist For Each Target
  • devops user exists
  • Sudoers entries added
  • Password auth on, AllowUsers devops
  • sshd restarted
  • Port 22 open from Jenkins

1. The Layered Config Model

There are three config files. At runtime, two configs are always live inside a commander. Each file has one clear job.

LayerFile / classHolds
Coreconfig.pyINI parsing, validation, file discovery, and DEFAULT inheritance. Nothing domain-specific.
Systemsystem_config.py SystemConfigPer-host SSH identity: user, sudo user, password, port. Always loaded.
Applicationpramata_config.py PramataConfigPer-host app secrets: git, Jenkins, WAF, GoDaddy, DB. Loaded per task.

2. config.py - The Core

This file turns INI files into validated Python dicts. It defines the file format, where config files live, and the shapes a config can take. It raises typed exceptions on bad sections, keys, or files.

It offers three access patterns. One host, many hosts, or a section per host with a [DEFAULT] fallback.

Two Axes Of Layering
  • Within one file: the [DEFAULT] section sets fleet-wide values, and each host section overrides only what differs.
  • Across files: the connection config and the task's secret config both live in the same commander at once.

3. system_config.py - Connection

This is the "how do I reach the box" layer. SystemConfig uses the section-per-host pattern. The [DEFAULT] section gives fleet-wide connection defaults, and each host section overrides only what changes. The engine feeds it straight into Fabric's SSH settings.

4. pramata_config.py - Secrets

This is the "once I am on the box, what secrets do I need" layer. PramataConfig uses the one-section-per-host pattern. It is not auto-loaded. A task opts into it, and the host is passed in for you so the right section is picked.

5. The Commander And Fabric

commander.py is the execution engine. It is a thin object-oriented wrapper over Fabric 1.x. A task declares a set of command generator methods. Each returns a list of command descriptor dicts. The engine walks those methods in source order and runs each descriptor through the matching Fabric operation.

  • Descriptors carry a cmd_type: sudo, run, local, put, get, or noop.
  • Connection settings come from system_config. Application secrets come from the task's chosen config.
  • Each command's return code is recorded and summed. The process exits with that code so Jenkins sees pass or fail.
  • print_only is dry-run. execute_all continues past failures. Both are first-class flags.

6. The Control Flow, End To End

From a Jenkins job to a Fabric call 
Jenkins / operator
  -> bin/execute_<task>.py        # parse options, build class, run()
     -> lib/<task>.py             # a Task(HostCommanderBase)
        - config_meta_data()      # names which config to load
        - command generators      # return lists of command dicts
     -> CommanderBase.run()       # walk generators in order
        -> execute_cmd()          # Fabric run / sudo / put / get / local
        -> sum return codes, sys.exit(code)
A newer pattern reads per-environment values from config/raas-<env>.yml with YAML, beside the original INI config. The codebase mixes eras. Both patterns are in use.

Add A New Config File

The pattern is small. One module in lib/, one .conf file in config/, and a one-line selector in the task that should use it. There is no central registry. Discovery happens by name at instantiation time.

Step 1: Create The Module Add lib/<name>_config.py. Subclass the access pattern that fits.
  • One host: HostBasedConfig
  • Many hosts: HostsBasedConfig
  • Section per host with defaults: SectionBasedConfig
Step 2: Create The File Add config/<name>.conf in the sibling config/ directory. For a host-based config, use one section per host, named exactly by the host string you pass in.
Step 3: Select It In A Task Point the task's config_meta_data() at your module and class. That return value is the switch that picks which config is active. Swap the module and class name there, and the task loads a different config, with no other change.

Minimal Template

A new config module for one host looks like this.

lib/example_config.py 
from config import HostBasedConfig


class ExampleConfig(HostBasedConfig):

    def config_file_name(self):
        return 'example.conf'

    def config_keys(self):
        return [
            'ssh_port',
            'user',
            'example_api_token',
            'example_api_secret',
        ]
config/example.conf 
[my-host.internal]
ssh_port           = 22
user               = devops
example_api_token  = REPLACE_ME
example_api_secret = REPLACE_ME
The one-line selector in a task 
class MyTask(HostCommanderBase):
    def config_meta_data(self):
        return {'module_name': 'example_config',
                'class_name': 'ExampleConfig'}
    # reach values via
    # self.config.get_config_value('example_api_token')
If the task extends HostCommanderBase and your class is host-based, the host is passed to it automatically. For fleet-wide defaults with per-host overrides, subclass SectionBasedConfig and also implement default_config_keys().

Fabric And OS Upgrade Pitfalls

Most breakage comes from an OS or Jenkins upgrade that pulls in Python 3 Fabric. The framework runs on Fabric 1.x, which needs Python 2.7. A Python 3 fab shim breaks it.

Symptom After an Ubuntu upgrade you see:
The tell-tale error 
ImportError: cannot import name 'isMappingType'
This comes from fabric/main.py running under Python 3. The root cause is a Python 3 fab shim over Fabric 1.x code.

Fix A: Keep Fabric 1.x On Python 2.7

This is the fix that matches how the framework is meant to run. Point fab back at Python 2.7 and reinstall Fabric 1.x under Python 2.

Restore Fabric 1.10.2 on Python 2.7 
# 1. point the fab shebang back to python 2.7
#    edit /usr/local/bin/fab
#    change  #!/usr/bin/python3  ->  #!/usr/bin/python2.7

# 2. remove the python3 fabric/paramiko packages
sudo apt-get remove 'python3.*'

# 3. install pip for python 2.7
curl https://bootstrap.pypa.io/pip/2.7/get-pip.py --output get-pip.py
sudo python2 get-pip.py

# 4. install Fabric 1.x
pip2 install fabric==1.10.2

# 5. if you hit: ImportError: No module named Crypto
sudo pip install pycrypto
Install Fabric for every user that runs it, not just devops. The tomcat user needed it too. The safest route is to install as root so it is on the path for all of them.

Fix B: Fabric3 On Python 3

A later working sequence used Fabric3 on Python 3 instead of restoring Fabric 1.x. Use this only if you have decided to move off Python 2.7.

The Fabric3 alternative 
sudo apt update
sudo apt install python3 python3-pip
sudo pip3 install "invoke>=1.3.0"
python3 -m pip install --upgrade pip setuptools wheel
pip3 install Fabric3
Quick Reference
  • isMappingType import error means a Python 3 fab over Fabric 1.x. Pick Fix A or Fix B.
  • No module named Crypto means pycrypto is missing. Install it.
  • Install Fabric for every user that runs it. Prefer installing as root.
  • Confirm the result with fab -V before running any job.
