1. What is Kubernetes?

Kubernetes (K8s) is an open-source container orchestration platform. It automates how containers are deployed, scaled, and networked. It also handles their day-to-day management. Google built it originally. The CNCF maintains it today.

In Simple Words Kubernetes runs our containerized applications reliably at scale.

2. Why Kubernetes Was Created

Earlier we ran containers directly with Docker. Managing many containers by hand became very hard. Kubernetes solves that problem for us.

Problems With Docker Alone
  • Hard to manage many containers
  • No automatic scaling
  • No self-healing when a container fails
  • No built-in load balancing
  • No rolling updates
  • Networking and storage are hard to wire up

3. Features of Kubernetes

  • Automatic deployment and scaling
  • Self-healing that restarts failed containers
  • Load balancing and service discovery
  • Rolling updates and easy rollback
  • Storage orchestration
  • Secret and configuration management
  • Runs anywhere, whether on-prem, cloud, or hybrid

4. Container Orchestration

Container orchestration means managing the full lifecycle of containers for us. That covers deploying, scaling, networking, and monitoring. Kubernetes does all of this automatically.

Docker vs Kubernetes

1. Kubernetes Architecture

Kubernetes follows a master-worker (client-server) architecture. The cluster splits into two main parts.

  • Control Plane (Master)
  • Worker Nodes
User / Client → kubectl → Kubernetes Cluster → Control Plane (Master) Worker Nodes (Minions)

2. Control Plane Components

The control plane makes global decisions about the cluster, such as scheduling. It watches for cluster events and responds to them. It keeps the cluster in the desired state.

  • API Server - the front end of Kubernetes. It exposes the REST API that kubectl and other components use.
  • etcd - a consistent, highly available key-value store that holds all cluster data.
  • Scheduler - watches for new Pods and picks a Node for each one to run on.
  • Controller Manager - runs the controllers that regulate cluster state, such as the Node and ReplicaSet controllers.

3. Worker Node Components

Worker Nodes run the containerized applications. They keep the runtime environment healthy.

  • kubelet - makes sure containers are running inside a Pod.
  • kube-proxy - maintains the network rules for communication.
  • Container Runtime - runs the containers, for example containerd or Docker.

1. What is a Pod?

A Pod is the smallest deployable unit in Kubernetes. It wraps one or more containers that share the same network and storage. Containers in a Pod are always scheduled together on the same node.

A Pod moves through several phases during its life.

Pendingaccepted, not yet running→ Runningbound to a node, containers up→ Succeededfinished, will not restart→ Failederror, will not restart→ Unknownstate cannot be read

4. Why a Pod Instead of a Container?

Container (Docker)Pod (Kubernetes)
Single containerOne or more containers
Manual managementManaged by Kubernetes

In short: a Pod is the smallest unit Kubernetes creates or deploys. It wraps one or more containers that are tightly coupled.

1. Deployments

A Deployment manages ReplicaSets for us. It gives declarative updates for Pods and ReplicaSets.

It helps with:

  • Scaling
  • Rolling updates
  • Rollback
  • Self-healing
Deploymentmanages ReplicaSet → ReplicaSetkeeps desired count → Pods the running copies

2. ReplicaSet

A ReplicaSet keeps a set number of Pod replicas running at all times. If a Pod fails, the ReplicaSet creates a new one to match the desired count.

Key Points
  • Ensures high availability
  • Maintains the desired number of Pods
  • Monitors Pods continuously
  • Works under a Deployment

3. Scaling

We can scale Pods up or down based on the load.

  • Manual scaling - use kubectl scale.
  • Auto scaling - use the Horizontal Pod Autoscaler (HPA).

4. Rolling Update

A rolling update lets us ship a new version without downtime. New Pods come up one by one while old Pods retire gradually.

5. Rollback

If an update goes wrong, we can roll back to the last stable version quickly.

6. kubectl Commands (Deployments)

Everyday deployment commands 
# list, create and scale
kubectl get deployments
kubectl create deployment myapp --image=nginx
kubectl scale deployment myapp --replicas=5

# roll out a new image, then undo if it goes wrong
kubectl rollout status deployment/myapp
kubectl set image deployment/myapp nginx=nginx:1.25
kubectl rollout undo deployment/myapp

1. Services - Why?

Pods are ephemeral. They can be created, destroyed, and recreated at any time. Their IP addresses keep changing. A Service gives a stable IP and DNS name to reach a group of Pods. It also load balances traffic across them.

Client → Servicestable IP / DNS → Pod Pod Pod

2. Service Types

TypeUseAccessible From
ClusterIPInternal communication inside the clusterInside the cluster. This is the default.
NodePortExpose the service on each Node's IPOutside the cluster, via Node IP and port
LoadBalancerExpose the service to external trafficOutside the cluster, via a cloud load balancer
ExternalNameMap the service to an external domainInside the cluster, resolved to a DNS name

3. Service Example (ClusterIP)

  • Service name: my-service
  • Type: ClusterIP
  • Cluster IP: 10.96.0.15
  • Port: 80, target port: 8080
A client inside the cluster reaches my-service:80. Traffic is load balanced across the Pods.

4. Try It With kubectl

Create and inspect a Service 
# expose a deployment as a ClusterIP service
kubectl expose deployment myapp \
  --port=80 --target-port=8080

# list and inspect services
kubectl get services
kubectl describe service my-service
