Module 01

What is ArgoCD?

The Kubernetes-native GitOps tool that makes Git your only deployment button — no more kubectl apply from CI pipelines

👶 ELI5

Imagine you have a security guard watching your Kubernetes cluster 24/7. He has a rulebook (your Git repo) that says exactly how the cluster should look. If someone sneaks in and manually changes a deployment — the guard notices the difference and puts it back. If you update the rulebook (push to Git) — the guard reads the new rules and updates the cluster automatically. That security guard is ArgoCD. Git is always boss.

😰 Before ArgoCD

Push-Based CD (The Problem)

# Jenkins/GitHub Actions pipeline: kubectl apply -f deploy.yaml # ← credentials in CI! kubectl apply -f service.yaml kubectl apply -f ingress.yaml
  • Cluster credentials stored in CI secrets
  • No drift detection — cluster can diverge from Git
  • Rollback = re-run old pipeline (if logs kept)
  • Who deployed what? Check CI logs... if they exist
  • Disaster recovery = run pipeline + pray
✅ With ArgoCD

Pull-Based GitOps (The Solution)

# Only command you need: git push origin main # ArgoCD detects → syncs → done ✓
  • Zero cluster credentials outside the cluster
  • Self-healing — drift reverted automatically
  • Rollback = git revert + push (90 seconds)
  • Full audit trail in Git — commit, author, PR, time
  • Disaster recovery = argocd sync (cluster rebuilt from Git)
🔄 DevOps Analogy

ArgoCD = Kubernetes' Desired State Enforcer

📁
GitSingle source of truth — like Terraform state, but for live K8s objects
🔄
ArgoCDThe reconciliation loop — constantly compares desired (Git) vs actual (cluster)
⚙️
Like systemdDefines what should run and restarts it if it drifts — but for your entire Kubernetes cluster
🏗️
Like Terraformterraform plan = argocd app diff | terraform apply = argocd app sync

Key Concepts

📱
Application
CRD that defines source (Git path) + destination (cluster/namespace). One app per microservice.
🗂️
Project
RBAC boundary and namespace isolation. Team A can't deploy to Team B's namespace.
🔁
Sync
The act of applying Git state to the cluster. Can be manual (prod) or automated (dev/staging).
❤️
Health
Is the deployed resource actually working? Healthy/Degraded/Progressing — separate from Sync status.
📦
Repository
Git repo, Helm registry, or OCI registry that ArgoCD monitors for changes.
🌳
App-of-Apps
Parent Application that manages child ArgoCD Applications in Git. Scale to 100 services.

Module 02

GitOps — The Four Principles

Why GitOps exists and how it fundamentally changes your entire deployment model

👶 ELI5

GitOps = Ops done through Git. Instead of clicking buttons or running CLI commands to deploy, you commit to Git. An agent inside your cluster reads Git and makes reality match it. Always. Automatically. If reality drifts, the agent corrects it.

1
Declarative

Describe what you want, not how to get there. Write replicas: 3 in a YAML file — not kubectl scale --replicas=3 in a script. K8s manifests, Helm values, Kustomize overlays — all declarative by nature.

# Declarative (GitOps ✓) spec: replicas: 3 image: myapp:v2.0.0 # Imperative (not GitOps ✗) kubectl scale deploy myapp --replicas=3 kubectl set image deploy/myapp app=myapp:v2.0.0
2
Versioned & Immutable

Git is the single source of truth. Every deployment change is a commit — immutable, timestamped, attributable. You can git log your deployments the same way you git log your code.

git log production/values.yaml # a3f9c12 alice bump image to v2.1.0 (2 hours ago) # b7e4d80 bob scale to 5 replicas (yesterday) # c9a1f35 alice add readinessProbe (3 days ago)
3
Pulled Automatically

The agent inside the cluster pulls changes from Git — not pushed from outside. The cluster reaches out to Git, not the other way around. No cluster credentials ever leave the cluster.

⬆️ PUSH (Old Way)
Developer
↓ git push
CI Pipeline
↓ kubectl apply
⚠️ cluster creds in CI!
K8s Cluster
⬇️ PULL (GitOps)
Developer
↓ git push
Git Repo
↓ ArgoCD polls
✅ creds inside cluster
K8s Cluster + ArgoCD
4
Continuously Reconciled

An automated agent runs forever — every 3 minutes by default — comparing desired state (Git) to actual state (cluster). Any drift is corrected. Self-healing is the default, not a feature you bolt on.

# Someone runs this manually at 3am: kubectl scale deploy payments-api --replicas=1 # ArgoCD reconciliation loop (next cycle, ~3 min): # Desired (Git): replicas: 3 # Actual (cluster): replicas: 1 # Action: kubectl apply → replicas restored to 3 # Alert: "OutOfSync detected and corrected"
🎬 Real Scenario

2:30 AM — Production down. 6 AM — your manager asks questions.

❌ Jenkins push pipeline:

Check CI logs — if not rotated. Hope the deploy step captured the image tag. The approver? Check Jira... maybe. The exact diff? Unavailable.

✅ GitOps with ArgoCD:

git log --since="yesterday" \ production/values.yaml # a3f9c12 alice@co.com # "chore: bump payments-api to v2.1.0" # PR #482, approved by: bob@co.com # 2026-03-17 23:47:33 UTC
💡 GitOps vs DevOps

GitOps is not a replacement for DevOps — it's the deployment part of DevOps done declaratively through Git. CI still builds and tests your code. GitOps takes over at the "deploy to cluster" step. Think of it as: CI builds the artifact, GitOps delivers it.

📁 GitOps Repository Patterns — Where Do All the Files Go?

One of the first questions every new ArgoCD user has: "Where do I put my ArgoCD Application YAMLs and Helm charts?" There are two main patterns:

📦 Pattern 1: Monorepo
App code + K8s manifests in the same repo. Simple for small teams.
sre-autopilot/             ← one repo
├── app/                   ← FastAPI code
│   ├── main.py
│   └── Dockerfile
├── helm/                  ← Helm chart
│   ├── Chart.yaml
│   ├── values.yaml
│   └── templates/
└── argocd/                ← ArgoCD apps
    ├── app-dev.yaml
    └── app-prod.yaml
✅ Simple · Easy to see code + config together
❌ ArgoCD re-syncs on every code commit, even non-K8s changes
🔀 Pattern 2: App-Config Repo (Recommended)
Separate repo for K8s configs. ArgoCD only watches this repo.
sre-autopilot/             ← app code repo
├── app/
│   └── main.py
└── Dockerfile

sre-autopilot-config/      ← config repo (ArgoCD watches this)
├── helm/
│   ├── Chart.yaml
│   ├── values.yaml
│   ├── values-dev.yaml
│   └── values-prod.yaml
├── argocd-apps/
│   ├── dev.yaml
│   └── prod.yaml
└── README.md
✅ Clean separation · ArgoCD syncs only on config change
✅ CI writes to config repo → ArgoCD picks up automatically
🏆 For the SRE Auto-Pilot project: Use Pattern 2. The app code repo has the Dockerfile. GitHub Actions builds the image, updates image.tag in values-prod.yaml in the config repo, commits, and pushes. ArgoCD watches the config repo and auto-deploys within 3 minutes.

Five components, one reconciliation loop — all running inside your cluster

👶 ELI5

ArgoCD is 5 containers running inside your cluster. One watches Git. One clones the Git repo and turns templates into real YAML. One compares that YAML to what's actually deployed. One handles the web UI and CLI. One handles who's allowed to log in. That's the whole system.

# 2. Apply official manifests

Module 04

ArgoCD Applications — The Core CRD

Every deployment starts with an Application YAML — the CRD that connects Git to your cluster

👶 ELI5

An ArgoCD Application is a YAML file that says: "Watch this Git repo at this path, and keep this Kubernetes namespace in sync with it." That's all. One Application = one microservice. ArgoCD reads this YAML and does all the work.

📱 Annotated Application YAML
YAMLpayments-api Application
apiVersion: argoproj.io/v1alpha1
kind: Application              # ArgoCD custom resource
metadata:
  name: payments-api
  namespace: argocd            # Always lives in argocd namespace
spec:
  project: production          # RBAC boundary (ArgoCD Project)

  source:
    repoURL: https://github.com/myorg/k8s-config
    targetRevision: HEAD       # Branch / tag / commit SHA
    path: apps/payments-api    # Folder with K8s manifests

    # --- For Helm charts, replace path with: ---
    # chart: payments-api
    # helm:
    #   valueFiles: [values.yaml, values-prod.yaml]

  destination:
    server: https://kubernetes.default.svc  # In-cluster target
    namespace: production

  syncPolicy:
    automated:
      selfHeal: true    # Revert manual kubectl changes
      prune: true       # Delete K8s resources removed from Git
    syncOptions:
      - CreateNamespace=true   # Auto-create namespace
      - ServerSideApply=true   # Use server-side apply
🔁 Sync Status

Does Git match the cluster?

✅ Synced — Git and cluster match perfectly
⚠️ OutOfSync — Cluster differs from Git
🔄 Syncing — Applying changes right now
❓ Unknown — Can't determine sync state
❤️ Health Status

Is the deployed app working?

🟢 Healthy — All resources running fine
🟡 Progressing — Rollout in progress
🔴 Degraded — Pod crashing / unavailable
⏸️ Suspended — App paused manually
🚫 Missing — In Git but not in cluster
💡 Critical Insight

You can be Synced but Degraded. The YAML was applied successfully, but the pod is CrashLoopBackOff. Sync ≠ Healthy. Always check both statuses.

🌳 App-of-Apps Pattern

Scale to 50 services without losing your mind

Problem: 30 microservices = 30 Application YAMLs to manage, update, and troubleshoot individually.

Solution: One "root" Application that watches a Git folder full of Application YAMLs — and creates them all automatically.

🌳 root-app (Application) ↓ watches /argocd-apps/ in Git
📱 payments-api.yaml (Application CRD)
📱 auth-service.yaml (Application CRD)
📱 frontend.yaml (Application CRD)
📱 ... 27 more services
# Git repo structure for App-of-Apps: argocd-apps/ ├── payments-api.yaml ← kind: Application ├── auth-service.yaml ← kind: Application ├── frontend.yaml ← kind: Application └── ... # Root Application just points here: spec.source.path: argocd-apps/ # ArgoCD creates all child Apps automatically
⚡ ApplicationSet — Next Level

Instead of writing 30 Application YAMLs, write ONE ApplicationSet with a template and a generator. ArgoCD creates all Applications automatically.

YAMLApplicationSet with git-directory generator
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: all-services
spec:
  generators:
  - git:
      repoURL: https://github.com/myorg/k8s-config
      revision: HEAD
      directories:
      - path: apps/*         # Every folder = one Application
  template:
    metadata:
      name: '{{path.basename}}'  # app name = folder name
    spec:
      source:
        path: '{{path}}'
        repoURL: https://github.com/myorg/k8s-config
      destination:
        server: https://kubernetes.default.svc
        namespace: production

Module 05

Sync Policies — Auto-Heal, Prune & Waves

Make your cluster self-healing — and know exactly when not to

🔁 Manual vs Auto-Sync
Manual SyncAuto-Sync
When to useProduction — human approval gates requiredDev / staging — deploy on every commit
Triggered byargocd app sync or UI clickArgoCD polls Git every ~3 min (or webhook)
selfHealDisabled — manual changes surviveEnabled — manual changes reverted
pruneDisabled — deleted Git resources keptEnabled — resources removed from Git get deleted
RiskDrift builds up if nobody runs syncPrune can delete resources if Git has a typo
🛡️ selfHeal: true

Someone runs kubectl scale or kubectl apply manually. ArgoCD detects the drift and reverts it on the next reconciliation cycle.

syncPolicy: automated: selfHeal: true # Result: # kubectl scale deploy myapp --replicas=1 # ArgoCD: "OutOfSync detected" # ArgoCD: "kubectl apply → replicas: 3 restored"
⚠️ prune: true

You remove a resource from Git (e.g., an old ConfigMap). ArgoCD deletes it from the cluster. Powerful — but dangerous if a YAML file has a typo.

syncPolicy: automated: prune: true # ⚠️ Warning: test prune in dev first. # If a valid manifest is accidentally # excluded from Git, ArgoCD will delete # the K8s resource from the cluster.
🔄 Sync Phases & Hooks
PhaseWhen it runsCommon use cases
PreSyncBefore any resources are appliedDB migrations, backups, schema validation
SyncMain resource apply (Deployments, Services, etc.)Everything else — the normal deploy
PostSyncAfter all resources are healthySmoke tests, cache warm-up, Slack notifications
SyncFailIf the sync failsAlert on-call team, trigger rollback procedure
YAMLPreSync DB migration hook
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migrate
  annotations:
    argocd.argoproj.io/hook: PreSync           # Run before apply
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
    argocd.argoproj.io/sync-wave: "-1"       # Before wave 0
spec:
  template:
    spec:
      containers:
      - name: migrate
        image: myapp:v2.0.0
        command: ["python", "manage.py", "migrate"]
🌊 Sync Waves — Ordering Resources

Within a sync phase, waves control order. Lower wave number applies first. Use to ensure namespaces and RBAC exist before workloads.

wave: -1Namespace creation
→
wave: 0RBAC + ServiceAccounts
→
wave: 1ConfigMaps + Secrets
→
wave: 2Deployments + Services
metadata: annotations: argocd.argoproj.io/sync-wave: "2" # Deploy after RBAC is ready
🚫 Sync Windows — Deploy Freezes

Block all syncs during maintenance windows, high-traffic periods, or change freezes.

YAMLNo deploys on Friday evenings
spec:
  syncWindows:
  - kind: deny
    schedule: "0 17 * * 5"    # Fridays 5pm
    duration: 3h
    applications:
    - "*"                    # All apps
    namespaces:
    - production
  - kind: allow
    schedule: "0 2 * * 1-5"   # Weekdays 2am
    duration: 2h              # Maintenance window
    applications:
    - "*"

🚨 When Things Go Wrong — ArgoCD Troubleshooting Guide

ArgoCD will show you what is wrong. Here's how to fix the most common problems a new user hits:

SYNC FAILED ComparisonError: failed to get cluster info
Cause: ArgoCD can't reach the target cluster. Usually a network/cert issue or the cluster was rebuilt.
argocd cluster list                    # Is the cluster registered?
argocd cluster get https://k8s.svc    # Show cluster connection status
kubectl get pods -n argocd             # Are ArgoCD pods running?
SYNC FAILED helm template failed: error converting YAML to JSON
Cause: Your Helm templates have a YAML syntax error. ArgoCD can't render the manifests.
# Test locally before pushing to Git:
helm template my-release ./helm-chart --values values-prod.yaml
# Also lint the chart:
helm lint ./helm-chart
DEGRADED Synced ✅ but Degraded 💔 — Pod CrashLoopBackOff
Cause: ArgoCD applied the manifests correctly, but the app is crashing. This is an app bug, not ArgoCD issue.
# Get pod details:
kubectl get pods -n production
kubectl describe pod <pod-name> -n production   # Events section
kubectl logs <pod-name> -n production --previous # Last crash logs
# In ArgoCD UI: click the Deployment → click the Pod → "Logs" tab
STUCK OUTOFSYNC App keeps flipping OutOfSync even after sync
Cause: An external controller (HPA, admission webhook, operator) is modifying the resource after ArgoCD applies it. Classic "drift loop."
# See the actual diff causing OutOfSync:
argocd app diff my-app
# Common fix: use ignoreDifferences in Application YAML:
spec:
  ignoreDifferences:
  - group: apps
    kind: Deployment
    jsonPointers:
    - /spec/replicas   # HPA manages replicas — ignore this field
SYNC FAILED ProgressDeadlineExceeded — rollout stuck
Cause: The new pods can't start within the deadline (600s default). New image broken, resource limits too low, or node doesn't have capacity.
# Roll back immediately while you investigate:
argocd app rollback my-app
# Check what happened to new pods:
kubectl describe deployment my-app -n production
kubectl get events -n production --sort-by='.lastTimestamp'
💡 Golden debug rule: Start in the ArgoCD UI → click the red/orange resource → read the Events tab → then go to kubectl. 90% of issues are visible in the UI without any CLI commands.

Module 06

RBAC, Projects & Security

Who can sync what, to where — and how to keep secrets out of Git

🗂️ ArgoCD Projects — Isolation Boundaries

An ArgoCD Project is a namespace isolation + access control boundary for a team. Team A's apps can only deploy to Team A's namespaces from Team A's repos — even if they have ArgoCD access.

YAMLAppProject for the payments team
apiVersion: argoproj.io/v1alpha1
kind: AppProject
metadata:
  name: payments-team
  namespace: argocd
spec:
  description: "Payments team — production namespace"

  # Only allow from these repos
  sourceRepos:
  - 'https://github.com/myorg/payments-*'

  # Only allow deploying to these namespaces
  destinations:
  - namespace: payments-production
    server: https://kubernetes.default.svc
  - namespace: payments-staging
    server: https://kubernetes.default.svc

  # Block dangerous cluster-level resources
  clusterResourceBlacklist:
  - group: ''
    kind: Node
  - group: rbac.authorization.k8s.io
    kind: ClusterRole
🔐 RBAC Policy Syntax
RBACargocd-rbac-cm ConfigMap
# Policy format: p, subject, resource, action, object, effect
# Subjects: role:name or user email or group
# Resources: applications, clusters, repositories, projects
# Actions: get, create, update, delete, sync, override, action

p, role:devops, applications, sync,   */*, allow
p, role:devops, applications, get,    */*, allow
p, role:devops, applications, delete, */*, deny    # No deletes!

p, role:readonly, applications, get,  */*, allow
p, role:readonly, applications, sync, */*, deny

# Assign users to roles
g, alice@company.com, role:devops
g, ops-team,          role:devops     # SSO group
g, bob@company.com,   role:readonly
🔒 Secrets — Never Store Plaintext in Git

The biggest ArgoCD mistake: putting password: mysecret in values.yaml and pushing to Git. Here are the production-safe alternatives:

ApproachToolHow it worksBest for
Vault PluginArgoCD Vault Plugin (AVP)Replaces <path:vault/data/prod#key> placeholders at sync time✅ HashiCorp Vault users
External SecretsESO + AWS SM / GCP SMK8s operator syncs secrets from cloud secret stores as K8s Secrets✅ Cloud-native teams
Sealed SecretsBitnami SealedSecretsEncrypt secrets with cluster-specific key, store encrypted YAML in Git✅ Simple, K8s native
SOPSMozilla SOPS + Age/KMSEncrypt YAML values in Git, decrypt at sync time✅ Helm chart teams
❌ Plaintextvalues.yamlSecrets visible in Git history forever❌ Never in production
🔑 SSO Setup — GitHub OAuth via Dex
YAMLargocd-cm ConfigMap — GitHub OAuth
apiVersion: v1
kind: ConfigMap
metadata:
  name: argocd-cm
  namespace: argocd
data:
  url: https://argocd.mycompany.com

  dex.config: |
    connectors:
    - type: github
      id: github
      name: GitHub
      config:
        clientID: $GITHUB_CLIENT_ID
        clientSecret: $GITHUB_CLIENT_SECRET
        orgs:
        - name: myorg           # Restrict to your GitHub org
          teams:
          - devops-team
          - platform-team
✅ Security Checklist
✅ Enable SSO (Dex + GitHub/OIDC) — disable local users in prod
✅ Create ArgoCD Projects per team — restrict source repos and destinations
✅ Use role:readonly for developers — only DevOps/SRE gets sync permission
✅ Never use the default project in production — always create named projects
✅ Use Sealed Secrets / ESO — no plaintext secrets in Git, ever
✅ Enable argocd-notifications for sync failures and degraded apps
✅ Set resource limits on ArgoCD components — it runs in prod cluster
⚠️ Rotate admin password after initial setup — or disable it entirely

Module 07 · Connecting the Dots

Helm + ArgoCD — Best of Both Worlds

Use ArgoCD to continuously deliver your Helm charts — no manual helm upgrade ever again

👶 Why this combination wins

Helm gives you templating power — one chart, many environments. ArgoCD gives you GitOps delivery — git push = deployed, drift-detected, audited. Together: every helm upgrade is a git commit, every rollback is a git revert, and your cluster is always as-declared-in-Git.

📱 ArgoCD Application — Helm Source
YAMLApplication pointing to a Helm chart
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: payments-api-prod
  namespace: argocd
spec:
  source:
    repoURL: https://github.com/myorg/k8s-config
    targetRevision: main
    path: charts/payments-api   # Helm chart directory in Git

    helm:

⚠️ Common Gotchas: Helm + ArgoCD
Release.Name mismatch: By default ArgoCD sets Release.Name = Application name. Always set helm.releaseName explicitly to avoid surprise name changes.
helm diff vs argocd app diff: The helm diff plugin doesn't work with ArgoCD's server-side rendering. Use argocd app diff myapp instead.
Hooks fire twice: If your Helm chart has hooks AND you configure ArgoCD sync hooks, both fire. Be careful of duplicate DB migrations or smoke tests.
helm test doesn't run: helm test commands don't execute via ArgoCD. Use a PostSync hook Job instead.
Sensitive values in inline values: The values: block in Application YAML is stored in the cluster as plaintext. Use Sealed Secrets or ESO for anything sensitive.

🤖 GitHub Actions → ArgoCD: The Complete CI/CD Pipeline

The most common real-world pattern: CI (GitHub Actions) builds and tags the image → writes the new tag to Git → ArgoCD detects the Git change → deploys automatically. Here's the exact workflow for the SRE Auto-Pilot:

Step 1 — CI: Build & Push Image (app code repo)
# .github/workflows/build.yml  (in sre-autopilot/ repo)
name: Build and Deploy
on:
  push:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build & push Docker image
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/myorg/sre-autopilot:${{ github.sha }}

      # This is the KEY step — update image tag in the CONFIG repo
      - name: Update image tag in config repo
        run: |
          git clone https://x-access-token:${{ secrets.CONFIG_REPO_TOKEN }}@github.com/myorg/sre-autopilot-config
          cd sre-autopilot-config
          # Use yq or sed to update the tag in values-prod.yaml
          yq e -i '.image.tag = "${{ github.sha }}"' helm/values-prod.yaml
          git config user.email "ci@myorg.com"
          git config user.name "CI Bot"
          git commit -am "ci: bump image tag to ${{ github.sha }}"
          git push
Step 2 — ArgoCD: Auto-detects the Git change (config repo)
ArgoCD polls every 3 min OR configure a webhook for instant detection
ArgoCD sees: values-prod.yaml changed → image.tag: abc123 → new version
App status → OutOfSync → ArgoCD runs: helm template → kubectl apply
Deployment rollout starts → pods updated → App status → Synced ✅ Healthy ✅
Step 3 — Webhook (optional but recommended for instant deploys)
# In sre-autopilot-config repo → Settings → Webhooks → Add webhook
Payload URL: https://your-argocd.company.com/api/webhook
Content type: application/json
Secret: (set same as argocd-secret webhook.github.secret)
Events: Just the push event ✅

# ArgoCD argocd-cm ConfigMap - enable webhook processing:
data:
  webhook.github.secret: "your-webhook-secret"

📄 Step 5 — ArgoCD Application YAML for prod (argocd-apps/sre-prod.yaml)
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: sre-autopilot-prod
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/myorg/sre-autopilot-config
    targetRevision: main
    path: helm                           # folder with the Helm chart

🔄 Step 7 — GitHub Actions: auto-update image tag on every merge to main
# .github/workflows/deploy.yml in sre-autopilot/ (app code repo)
name: Build & GitOps Deploy
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build image
        run: docker build -t ghcr.io/myorg/sre-autopilot:${{ github.sha }} .
      - name: Push image
        run: docker push ghcr.io/myorg/sre-autopilot:${{ github.sha }}
      - name: Update config repo image tag
        run: |
          git clone https://x-access-token:${{ secrets.CONFIG_TOKEN }}@github.com/myorg/sre-autopilot-config
          cd sre-autopilot-config
          # Update the image tag — ArgoCD will detect this Git change and deploy
          sed -i 's|tag: ".*"|tag: "${{ github.sha }}"|' helm/values-prod.yaml
          git commit -am "ci: deploy ${{ github.sha }}"
          git push
